## Recommended solution: add a new default `0:Workflow` tab

The best design is **not** to overload the existing `Tests` tab. In this repo, `Tests` currently means historical VTI/bottleneck intelligence, while `Jobs` is the live runner/pipeline view. The requested feature is a third thing: a **test execution workflow graph** that answers:

> “Given this change, what tests and gates are required, what can run in parallel, what must wait, and what is each item doing right now?”

So I would add a new top-level tab:

```text
[0:Workflow]  1:Mission  2:Release  3:Jobs  4:Agents  5:Tests  6:Pools  7:Cache  8:Evidence  9:Secrets  10:Git
```

`0:Workflow` should become the default active tab in `src/tui/app.rs`.

# Status colors

The user specifically asked for different colors for:

* running
* ran
* error
* waiting

I would use the repo’s existing color conventions but improve them slightly for this view.

| Workflow status | GitLab / DB status mapping                    | Color              | Icon               | Meaning                                 |
| --------------- | --------------------------------------------- | ------------------ | ------------------ | --------------------------------------- |
| Waiting         | `created`, `pending`, dependency not complete | Yellow or DarkGray | `○`                | Queued or blocked behind previous stage |
| Running         | `running`                                     | Cyan/Blue          | `●` / animated `◉` | Actively executing                      |
| Ran             | `success`, `passed`                           | Green              | `✓`                | Finished successfully                   |
| Error           | `failed`, `timeout`, `error`                  | Red                | `✗`                | Failed or errored                       |
| Canceled        | `canceled`                                    | DarkGray           | `⊘`                | Canceled                                |
| Skipped         | `vti-skipped`, `omitted`, docs-only no-op     | Gray/Green         | `↷`                | Intentionally not required              |
| Blocked         | dependency failed                             | Magenta/Red        | `◆`                | Cannot run because prerequisite failed  |

Important detail: I would treat **“ran”** as a terminal successful state, not merely “started once.” If a test has completed with failure, it should be **error**, not ran.

# Dependency and layout algorithm

The layout should use real DAG ranking:

```rust
stage(node) = 0 if node has no dependencies
stage(node) = 1 + max(stage(dep)) otherwise
```

Then:

```rust
nodes with same stage index are rendered side by side
nodes with greater stage index are rendered below
```

That gives the exact visual behavior requested.

# Rendering design

## Card anatomy

Each test block should use a consistent visual structure:

```text
┌─ ● Library Tests ───────────────┐
│ cargo nextest --lib -E tui      │
│ running 01:14 / ETA ~02:20      │
│ subsystem: tui  conf: 0.92      │
└─────────────────────────────────┘
```

A card should contain:

1. status icon
2. test/job name
3. short command
4. elapsed time or duration
5. reason/subsystem
6. optional badges:

   * `VTI`
   * `required`
   * `sentinel`
   * `critical`
   * `cached`
   * `blocked`

## Stage headers

Each stage should have an aggregate status:

```text
┌─ Stage 1 · Parallel test lanes · 1 running · 2 waiting · 0 failed ────────┐
```

Aggregate status rules:

```text
failed/error present   => stage error
running present        => stage running
waiting present        => stage waiting
all ran/success        => stage ran
all skipped            => stage skipped
```

## Animated running cards

Use `app.tick_count` to animate the icon:

```rust
let frames = ["●", "◉", "◎", "◉"];
let icon = frames[(app.tick_count as usize / 2) % frames.len()];
```

For running tests, the card border can alternate between cyan and blue.

## Critical path

Compute the longest path using historical durations from `test_executions` / `ci_job_runs`.

Highlight the critical path with bold borders:

```text
[CRIT]
```

This is useful because in a parallel graph the slowest dependency chain is what determines completion time.

# Navigation model

This is crucial. The view should not behave like a flat list.

## Arrow keys

```text
← / h     move to previous sibling in the same stage
→ / l     move to next sibling in the same stage
↑ / k     move to the closest node in the previous stage
↓ / j     move to the closest node in the next stage
```

“Closest” should preserve horizontal position. If the user is on the third card in Stage 1 and presses down, select the third card in Stage 2 if it exists; otherwise select the nearest available card.

## Stage navigation

```text
Home       first node in current stage
End        last node in current stage
PageUp     previous stage
PageDown   next stage
g          top of workflow
G          bottom of workflow
```

## Workflow actions

```text
Enter      open logs/inspector for selected test
Space      pin/unpin selected test inspector
r          retry selected failed/canceled job
c          cancel selected running job
f          jump to first failed node; if none, first running node
/          search tests/jobs/commands/reasons
n          next search result
N          previous search result
?          help
0          return to Workflow tab from anywhere
Tab        cycle tabs
Shift+Tab  reverse cycle tabs
F5         refresh now
```

The selected node should survive refreshes using stable IDs, not list indexes.

# Workflow collector

Add a collector similar to the existing flow collector:

```text
src/tui/workflow/collector.rs
```

It should run every 1.5 seconds, same cadence as the general TUI state sync.

Pseudo-code:

```rust
pub async fn run_collector(
    db: Db,
    gitlab: GitlabClient,
    tx: mpsc::Sender<TestWorkflowSnapshot>,
) {
    let mut interval = tokio::time::interval(Duration::from_millis(1500));

    loop {
        interval.tick().await;

        let plan = db.latest_test_plan(DEFAULT_RELEASE_PROJECT_ID).await?;
        let jobs = db.recent_job_events(200).await?;
        let pipelines = db.list_tracked_pipelines(10).await?;

        let snapshot = workflow::builder::build_snapshot(
            plan,
            jobs,
            pipelines,
            maybe_live_gitlab_jobs,
        );

        if tx.send(snapshot).await.is_err() {
            break;
        }
    }
}
```

Then add a workflow channel to `App`:

```rust
workflow_rx: mpsc::Receiver<TestWorkflowSnapshot>,
workflow_tx: mpsc::Sender<TestWorkflowSnapshot>,
```

And in `tick()`:

```rust
while let Ok(workflow) = self.workflow_rx.try_recv() {
    self.apply_workflow_snapshot(workflow);
}
```

Important: apply snapshots the same way the flow snapshot currently prevents blanking. If the new snapshot is empty but the previous one was non-empty, keep the previous one and mark it stale.

# Status overlay logic

The builder needs to match planned nodes to live jobs.

## Matching priority

For each workflow node:

1. exact GitLab job name match
2. generated VTI job name match
3. command substring match
4. pipeline/job artifact metadata match
5. fallback to DB `test_executions`

For internal selected VTI jobs, `src/test_intel/ci_gen.rs` currently generates names like:

```rust
vti-{subsystem}-{index}
```

The helper `sanitize_job_name()` is private. I would expose a stable helper:

```rust
pub fn selected_test_job_name(test: &SelectedTest, index: usize) -> String
```

Then both CI generation and workflow building use the same naming function.

For external jobs, the names are already real CI job names, so matching is straightforward.

# Required API improvement

I would add a read-only command that produces the exact same structure the TUI uses:

```bash
jeryu test workflow \
  --base origin/main \
  --head HEAD \
  --profile auto \
  --json
```

Possible flags:

```text
--base <ref>
--head <ref>
--repo-root <path>
--project-id <id>
--profile auto|inner-loop|pull-request|release-blocking|scheduled-hardening
--external-workspace <path>
--json
--emit-plan <path>
--emit-dot <path>
```

This matters because the TUI should not be the only way to inspect the workflow. It also makes this testable in CI.

Internally:

```rust
TestCommands::Workflow { ... }
```

would call:

```rust
workflow::builder::build_from_repo_diff(...)
```

and print a `TestWorkflowSnapshot`.

# How “ALL tests” should be shown on small terminals

A terminal cannot always physically show every card at once, especially for external workspaces with dozens of jobs. The design should still include all tests in one workflow model, with:

1. vertical scrolling
2. horizontal scrolling within wide stages
3. a mini-map strip
4. stage summary counts
5. search
6. jump-to-failed/running

For large terminals, the goal should be to show the entire active plan at once. For smaller terminals, nothing should disappear into another tab; it should remain part of the same navigable workflow canvas.

# My strongest recommendation

Build the new view as a **Workflow DAG**, not as another list and not as a small extension to the existing Jobs tab.

The architecture should be:

```text
VTI/TestMap/Proof lanes
        │
        ▼
Workflow builder
        │
        ▼
TestWorkflowSnapshot
        │
        ├── TUI 0:Workflow tab
        └── jeryu test workflow --json
```

The existing `Jobs` tab should remain the operational live runner view. The existing `Tests` tab should remain the historical test intelligence/bottleneck view. The new `0:Workflow` tab becomes the default, high-level visual control plane for required tests.

That gives the user exactly what they asked for:

* all needed tests
* correct order
* parallel tests side by side
* dependent tests below
* live status colors
* excellent keyboard navigation
* a default `0:Workflow` tab
* a Rust TUI experience that is much more impressive than a table of jobs.
