I studied the repo and the existing API/TUI shape. My strongest recommendation is: **add a new default `0:workflow` tab as a first-class validation DAG view**, rather than trying to overload the existing `Tests` or `Jobs` tabs.

I also made a screenshot-style mockup of the proposed rendering: [workflow TUI mockup](sandbox:/mnt/data/jeryu_workflow_tui_mockup.png).

## Executive recommendation

Build a new **Workflow** tab that answers one question better than any existing view:

> “What validation/tests are required, what can run in parallel, what must wait, and what is the current state of every block?”

The view should be a **top-to-bottom dependency flow**:

* **Parallel work** appears **side by side** in the same row.
* **Sequential dependencies** appear **below** their prerequisites.
* Every block is color-coded by status:

  * waiting / queued
  * running
  * passed / ran
  * failed / error
  * blocked
  * skipped / omitted
* The selected block opens an inspector with command, reason, dependencies, dependents, ETA, logs, evidence, and retry/cancel controls.
* `0:workflow` should be the **default tab** when the TUI opens.

The repo already has much of what this needs: test planning, test plan persistence, CI job discovery, flow graph machinery, status color helpers, and background refresh loops. What is missing is the **workflow-specific model, collector, layout algorithm, widget, navigation, and tab integration**.

## Existing test APIs are exactly the right data source

The API already has a strong test intelligence surface:

```text
jeryu test run
jeryu test plan
jeryu test batch
jeryu test select
jeryu test select-external
```

Important parts:

* `jeryu test run` runs a command through ephemeral GitLab CI and records execution.
* `jeryu test plan` infers risk, tags, timeout, and rationale.

* `jeryu test select` performs VTI smart test selection and can emit plan JSON and GitLab child pipeline YAML.

That means the new Workflow tab should not invent a new planning concept. It should visualize the **resolved validation plan** produced by these APIs, plus live CI execution state.

# What “needed tests” should mean

The phrase “ALL of the needed tests” needs a precise definition in code.

I recommend this definition:

A test/workflow node is **needed** when at least one of these is true:

1. It was selected by the current VTI plan.
2. It is a dependency of a selected test/job.
3. It is required by the active proof lane profile.
4. It is currently running in the active pipeline.
5. It failed in the active pipeline and is required or blocking.
6. It is a required sentinel/safety test for the current change type.

A test/job should be shown as **omitted/skipped** only when the planner explicitly marks it as safe to skip. Skipped tests should not clutter the main DAG by default, but they should be visible in a collapsible “omitted by selector” rail or inspector section.

This is important because the user asked for “ALL of the needed tests,” not every possible test in the repo.

## 2. Proof lanes

The repo has a machine-readable `proof-lanes.toml`.

The file says agents use it to determine validation commands, and it contains lanes such as:


So the Workflow tab should parse `proof-lanes.toml` and use it as a fallback/augmentation source when no live VTI plan exists.

Example:

```text
Changed src/tui/*:
  check
  unit
  integration or full, depending change type/risk
```

This is exactly the kind of deterministic validation plan users expect to see.

# Layout algorithm

The layout should be deterministic, stable, and visually meaningful.

## Step 1: collect candidate nodes

Sources:

```text
latest_test_plan
proof-lanes.toml
active GitLab pipeline jobs
recent test_executions
external testmap jobs
```

Each node gets a stable ID:

```rust
hash(project_id, plan_id, command_or_job_name)
```

Deduplicate by normalized command/job name.

For example, these should resolve to one logical node when appropriate:

```text
cargo nextest run -p jeryu --lib
test-lib
unit
```

but only if the mapping is known. Otherwise keep them separate and show source labels.

---

## Step 2: add dependency edges

Preferred edge order:

1. Exact `needs` / external testmap dependency.
2. Proof-lane dependency.
3. GitLab stage order.
4. Inferred fallback order.

The external testmap already contains useful dependency knowledge. For example, `testmap.rs` has logic where jobs like `test-rust-nextest-*` depend on `compile-workspace`, and jobs like `test-local-built` / `publish-rc-dry-run` depend on `build-enclave-server`. The materialized external GitLab YAML logic is in `src/test_intel/testmap.rs`.

For proof lanes, the dependency shape should be something like:

```text
check
  ├─ unit
  ├─ security/audit
  └─ format/lint, when applicable

unit
  └─ integration

integration
  └─ full
```

This does not mean the commands must always execute that way; it means the visualization should show the **safe validation order** unless an exact plan says otherwise.

---

## Step 3: topological sort into rows

Pseudo-code:

```rust
fn layout_levels(nodes: &BTreeMap<NodeId, Node>, edges: &[Edge]) -> Vec<WorkflowLevel> {
    let mut indegree = compute_indegree(nodes, edges);
    let mut ready = nodes
        .keys()
        .filter(|id| indegree[*id] == 0)
        .cloned()
        .collect::<Vec<_>>();

    let mut levels = Vec::new();

    while !ready.is_empty() {
        ready.sort_by(|a, b| compare_nodes_for_display(&nodes[a], &nodes[b]));

        let current = ready;
        levels.push(WorkflowLevel {
            index: levels.len(),
            label: format!("phase {}", levels.len()),
            nodes: current.clone(),
        });

        ready = Vec::new();

        for node_id in current {
            for child in dependents_of(node_id, edges) {
                indegree[child] -= 1;
                if indegree[child] == 0 {
                    ready.push(child);
                }
            }
        }
    }

    levels
}
```

Sorting within a level should be:

1. failed/error first
2. running second
3. critical path nodes
4. required before optional
5. longer estimated duration before shorter
6. stable label sort

That gives users the most actionable information first.

---

## Step 4: detect blocked nodes

A node is blocked when:

```text
status is waiting/queued
AND at least one dependency is error/canceled/blocked
AND this node is required
```

Blocked nodes should be visually distinct from merely waiting nodes.

---

## Step 5: compute critical path

Use historic median duration or live ETA:

```rust
weight(node) = eta_secs.or(median_secs).unwrap_or(default_secs)
```

Then compute longest weighted path through the DAG.

Display:

```text
critical path: check → unit → integration → full
estimated remaining: 14m20s
```

Critical nodes get a stronger border or marker.

# Navigation controls

The user specifically asked for great controls and easy up/down/left/right navigation.

Recommended controls:

```text
0                 workflow tab
1-9               other tabs
g                 git tab
← / h             previous parallel block in same row
→ / l             next parallel block in same row
↑ / k             nearest prerequisite row, preferably an actual dependency
↓ / j             nearest dependent row, preferably an actual dependent
Enter             open selected node logs/details
Space             collapse/expand phase or skipped group
Tab               next major pane: graph → inspector → logs
Shift+Tab         previous major pane
/                 search tests/jobs/commands
n / N             next/previous search match
r                 retry selected failed block
R                 retry failed required blocks
c                 cancel selected running block
f                 follow live log
e                 open evidence for selected block
?                 help overlay
q / Esc           quit/back
```

## Navigation should be graph-aware

Left/right is easy: move within the current topological level.

Up/down should not be merely screen-position based. It should prefer actual dependency relationships.

Algorithm:

```text
Up:
  1. If selected node has dependencies, select the dependency closest to current x.
  2. Else select nearest node in previous level by x position.

Down:
  1. If selected node has dependents, select the dependent closest to current x.
  2. Else select nearest node in next level by x position.
```

This makes the controls feel intelligent.

Example:

```text
check
  ↓
unit
  ↓
integration
```

Pressing down from `check` should select `unit`, not simply the first block in the next row.

Running:


Waiting:

```text
┌─○ integration──────┐
```

```text
╭─✗ clippy───────────╮
```

Blocked:

```text
╭─! full─────────────╮
│ blocked by clippy  │
```

Simple vertical edges are enough for v1:

```text

        ▼
```

A more advanced version can use Ratatui `Canvas` or direct buffer drawing for connector lines. But even normal `Span`/`Line` rendering can look excellent if the blocks are aligned and edges are stable.

For TUI changes, `proof-lanes.toml` maps `tui/` to `leaf-bugfix` at `proof-lanes.toml:97`. For test intelligence changes, it maps `test_intel/` to `api-change` at `proof-lanes.toml:91`.

# Status reconciliation

A node can have status from multiple sources:

```text
planner says selected
state says previous execution passed
GitLab says currently running
GitLab says current pipeline failed
```

Use this priority order:

1. Live GitLab status for active pipeline.
2. Active `test_executions` status.
3. Current plan status, if stored.
4. Dependency-derived status, such as blocked.
5. Default waiting.

Example:

```rust
fn reconcile_status(node: &mut WorkflowNode, live_job: Option<&Job>) {
    if let Some(job) = live_job {
        node.status = WorkflowStatus::from_gitlab(&job.status);
        return;
    }

    if node.deps.iter().any(|dep| dep_is_failed_or_blocked(dep)) {
        node.status = WorkflowStatus::Blocked;
        return;
    }

    if node.status == WorkflowStatus::Unknown {
        node.status = WorkflowStatus::Waiting;
    }
}
```

# The most important design decision

Do **not** make Workflow just another CI job table.

The requested experience is not:

```text
list of jobs
list of tests
pipeline stages
```

It is:

```text
a dependency-aware validation map
```

So the heart of the implementation should be:

```text
test plan + proof lanes + materialized dependencies + live job status
        ↓
normalized workflow DAG
        ↓
topological levels
        ↓
side-by-side parallel blocks
        ↓
graph-aware keyboard navigation
```

That will give the user exactly what they asked for: an incredible Rust TUI workflow view that makes the validation order obvious at a glance.
