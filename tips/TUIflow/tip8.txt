## What I found in the repo

The current TUI already has strong building blocks, but the exact view you want should be a **new first-class Workflow tab**, not a tweak to the existing Jobs or Tests tabs.

Right now:

* The default tab is `Mission`, not workflow. `ActiveTab::default()` returns `Mission` in `src/tui/app.rs:51-55`.
* Numbered tabs currently start at `1`, and `0` is effectively used for `10:Git` through the key handling in `src/tui/mod.rs:794-802` and `ActiveTab::from_number(10)` in `src/tui/app.rs:33-48`.
* The existing `Tests` tab is focused on bottlenecks and historical duration data, not the active Рђюwhat must run nowРђЮ graph. See `draw_tests()` in `src/tui/ui.rs:2066-2180`.
* The existing Jobs view is already visually richer, with a live runner feed, pipeline progress, matrix, and inspector in `src/tui/ui.rs:1186-1773`.
* There is an older flow graph system under `src/tui/flow/`, with `FlowSnapshot`, `PipelineFlow`, `FlowGraph`, and `FlowNode` in `src/tui/flow/model.rs`. It is useful, but it is **column-oriented**, not dependency-tier-oriented. It groups jobs by inferred pipeline columns such as build/tests/security/release, but it does not yet produce the exact Рђюparallel tests side-by-side, dependent tests belowРђЮ layout you described.
* The API already has the pieces needed to know what tests are required:

  * VTI/internal planner: `src/test_intel/planner.rs`
  * external testmap planner: `src/test_intel/testmap.rs`
  * GitLab job/pipeline APIs: `src/gitlab_client.rs`

  * test CLI/API docs: `docs/API.md`
  * proof lane routing: `proof-lanes.toml`

The best solution is to add a new default tab:

```text
0:Workflow
```

This tab becomes the command-center view for every required test/proof job, rendered as a dependency DAG where each row is a sequential step and each rowРђЎs cards are parallel work.

# Best solution: a default `0:Workflow` tab

The new tab should answer this question immediately:

> РђюWhat tests/proof jobs are required for this change, what order must they run in, which ones can run in parallel, and what is happening right now?РђЮ

It should not replace the existing Jobs tab. The Jobs tab remains the operational log/matrix view. The Workflow tab becomes the high-level orchestration map.

The tab order should become:

```text
0:Workflow  1:Mission  2:Release  3:Jobs  4:Agents  5:Tests  6:Pools  7:Cache  8:Evidence  9:Secrets
```

`Git` can remain available through the command palette, a named capture tab, or a non-number shortcut such as `g`. The user explicitly wants `0:workflow`, so `0` should no longer mean Git.

The default startup state should be:

```rust
impl Default for ActiveTab {
    fn default() -> Self {
        Self::Workflow
    }
}
```

And the number mapping should include:

```rust
0 => Some(Self::Workflow)
```

# What Рђюall needed testsРђЮ should mean

The Workflow tab should not blindly show every test that exists. It should show every test/proof job that is **required for the current plan**.

The source of truth should be selected in this priority order:

## 1. Live GitLab pipeline, when one exists

When there is an active tracked pipeline, release pipeline, or test-run pipeline, the Workflow tab should use actual GitLab jobs as the primary truth.

This gives real statuses, job IDs, traces, runner data, durations, and retry/cancel ability.

Use:

* `GitlabClient::list_pipeline_jobs_with_downstream`
* `GitlabClient::list_pipeline_jobs`
* `GitlabClient::get_pipeline`
* `GitlabClient::job_trace`
* `GitlabClient::retry_job`
* `GitlabClient::cancel_pipeline` / job-level cancel where applicable

The current flow collector already does something close to this in `src/tui/flow/collector.rs:119-183`.

## 2. Latest stored VTI test plan, when no live pipeline exists

The DB already stores test plans and test plan items:

* `test_plans`
* `test_plan_items`
* `test_executions`
* `selector_misses`

The current state API has:

```rust
latest_test_plan(project_id)
record_test_plan(...)
record_test_plan_item(...)
```

See `src/state.rs:2957-3090`.

The Workflow tab should add a richer read method:

```rust
latest_test_plan_full(project_id) -> Option<TestPlanRecord>
```

Where `TestPlanRecord` includes the plan metadata, payload, and items.

This lets the tab show the planned required tests before they have become GitLab jobs.

## 3. Planner-derived plan for current diff, as a fallback

When there is no active pipeline and no stored plan, the Workflow tab can construct a read-only preview from the current working diff using the internal planner:

```rust
test_intel::planner::plan_tests(changed_paths)
```

This is especially useful for local development.

## 4. Proof lanes from `proof-lanes.toml`

The repo already has proof routing:

```toml
[lane.check]
command = "cargo check -p jeryu --message-format=json"

[lane.unit]
command = "cargo nextest run -p jeryu --lib"

[lane.integration]
command = "cargo test -p jeryu --test '*' -- --test-threads=1"

[lane.full]
command = "cargo nextest run -p jeryu"

[lane.security]
command = "cargo test -p jeryu -- secrets exec honeypot admission"
```

The Workflow tab should include these proof lanes when they are required by the change type. For example:

```text
leaf-bugfix       -> check + unit
state-change      -> check + unit + integration
api-change        -> check + unit + integration
cross-module      -> check + unit + integration + full
release-blocking  -> check + unit + integration + full
security-relevant -> check + unit + integration + security
```

This matters because the user said **ALL needed tests**. In this repo, РђюneededРђЮ is not only GitLab jobs. It also includes proof-lane requirements.

# The core visual model

The Workflow tab should render a dependency graph as **horizontal tiers**.

Each tier is one required ordering level.

* Tests/jobs in the same tier can run in parallel.
* Tests/jobs in lower tiers must wait for dependencies above them.
* Each test/job is rendered as a card/block.
* The selected block gets a highlighted border.
* Connectors show dependency flow from upper blocks to lower blocks.

# Status colors

Use exactly the four primary user-facing statuses:

| Status    | Meaning                                                     | Color / style                                                 |
| --------- | ----------------------------------------------------------- | ------------------------------------------------------------- |
| `waiting` | planned but not runnable yet, or pending/created            | yellow border, dim body                                       |
| `running` | actively executing                                          | cyan/blue border, animated spinner, live progress             |
| `ran`     | completed successfully or intentionally skipped             | green border; skipped can be dim green/gray with `SKIP` badge |
| `error`   | failed, canceled unexpectedly, blocked by failed dependency | red border, bold error badge                                  |

I would still keep secondary internal statuses, but map them into these four for the main block color.

Example mapping:

```rust
pub enum WorkflowStatus {
    Waiting,
    Running,
    Ran,
    Error,

    // Optional secondary states for inspector/filtering.
    Skipped,
    Blocked,
    Unknown,
}
```

Main display mapping:

```rust
impl WorkflowStatus {
    fn display_status(&self) -> DisplayStatus {
        match self {
            WorkflowStatus::Waiting => DisplayStatus::Waiting,
            WorkflowStatus::Running => DisplayStatus::Running,
            WorkflowStatus::Ran => DisplayStatus::Ran,
            WorkflowStatus::Skipped => DisplayStatus::Ran,
            WorkflowStatus::Error => DisplayStatus::Error,
            WorkflowStatus::Blocked => DisplayStatus::Error,
            WorkflowStatus::Unknown => DisplayStatus::Waiting,
        }
    }
}
```

The current UI already has status color helpers in `src/tui/ui.rs:17-57`. The Workflow tab should reuse the same design language, but with stronger block-level color.

# Graph construction

The workflow graph should be constructed in the collector/background layer, not during rendering.

The TUI rendering code should receive a ready-to-draw `WorkflowSnapshot`.

## Step 1: find the source

The collector should run every 1.5 seconds, similar to the existing flow collector.

Pseudo-flow:

```rust
pub async fn run_workflow_collector(
    db: Db,
    gitlab: Option<GitlabClient>,
    tx: watch::Sender<WorkflowSnapshot>,
) {
    loop {
        let snapshot = build_workflow_snapshot(&db, gitlab.as_ref()).await
            .unwrap_or_else(|err| WorkflowSnapshot::error_empty(err));

        let _ = tx.send(snapshot);

        tokio::time::sleep(Duration::from_millis(1500)).await;
    }
}
```

Build priority:

```rust
async fn build_workflow_snapshot(
    db: &Db,
    gitlab: Option<&GitlabClient>,
) -> anyhow::Result<WorkflowSnapshot> {
    if let Some(live) = try_live_pipeline_workflow(db, gitlab).await? {
        return Ok(live);
    }

    if let Some(plan) = try_latest_test_plan_workflow(db).await? {
        return Ok(plan);
    }

    if let Some(preview) = try_current_diff_workflow(db).await? {
        return Ok(preview);
    }

    Ok(WorkflowSnapshot::empty())
}
```

## Step 2: build nodes

There are four important builders.

### A. Live GitLab pipeline builder

This uses actual job data.

Each GitLab job becomes a `WorkflowNode`.

Status mapping:

```rust
fn workflow_status_from_gitlab(status: &str) -> WorkflowStatus {
    match status {
        "success" => WorkflowStatus::Ran,
        "failed" => WorkflowStatus::Error,
        "running" => WorkflowStatus::Running,
        "pending" | "created" | "waiting_for_resource" | "preparing" => WorkflowStatus::Waiting,
        "canceled" => WorkflowStatus::Error,
        "skipped" | "manual" => WorkflowStatus::Skipped,
        _ => WorkflowStatus::Unknown,
    }
}
```

Dependency construction should prefer real GitLab `needs` if available. The current `Job` struct in `src/gitlab_client.rs` has `id`, `name`, `status`, `stage`, `duration`, runner, etc., but not `needs`. I would extend it:

```rust
#[derive(Debug, Clone, Deserialize)]
pub struct JobNeed {
    pub name: String,
    #[serde(default)]
    pub optional: bool,
}

#[derive(Debug, Clone, Deserialize)]
pub struct Job {
    // existing fields...
    #[serde(default)]
    pub needs: Vec<JobNeed>,
}
```

Then:

* If a job has `needs`, draw only those dependency edges.
* If no `needs` are present, fall back to GitLab stage ordering:

  * all jobs in stage N are parallel
  * all jobs in stage N+1 depend on all non-allow-failure required jobs in stage N

This fallback is important because GitLab configurations may not expose full `needs` through the existing API path.

# Tiering algorithm

Once nodes and edges are known, the layout should compute a topological rank.

Simple version:

```rust
fn assign_tiers(nodes: &[WorkflowNode], edges: &[WorkflowEdge]) -> Vec<WorkflowTier> {
    let mut rank = HashMap::<WorkflowNodeId, usize>::new();

    for node in topological_sort(nodes, edges) {
        let parent_rank = incoming_edges(&node.id, edges)
            .filter_map(|edge| rank.get(&edge.from))
            .copied()
            .max()
            .unwrap_or(0);

        let node_rank = if has_incoming_edges(&node.id, edges) {
            parent_rank + 1
        } else {
            0
        };

        rank.insert(node.id.clone(), node_rank);
    }

    group_nodes_by_rank(nodes, rank)
}
```

This gives the required behavior:

* same rank = can run side-by-side
* lower rank = must run after something above

For better visual stability, sort within a tier by:

1. status priority: `error`, `running`, `waiting`, `ran`
2. critical path/current blocker
3. stage order
4. original plan order
5. label

That keeps important cards near the left side where users naturally look first.

# Controls

The controls should make the graph feel great to navigate.

| Key        | Workflow behavior                                      |
| ---------- | ------------------------------------------------------ |
| `0`        | jump to Workflow tab                                   |
| `Tab`      | cycle top-level tabs                                   |
| `Рєљ` / `Рєњ`  | move to previous/next block in the same dependency row |
| `РєЉ` / `РєЊ`  | move to nearest block in the previous/next row         |
| `Enter`    | open logs/details for selected test/job                |
| `Space`    | toggle compact/full inspector or log preview           |
| `f`        | follow current running test/job                        |
| `e` or `!` | jump to first/next error                               |
| `[` / `]`  | previous/next node with same status                    |
| `/`        | search tests/jobs                                      |
| `r`        | retry selected failed job, when backed by GitLab       |
| `c`        | cancel selected running job, when supported            |
| `Home`     | jump to first required node                            |
| `End`      | jump to current blocker or final gate                  |
| `?`        | show workflow help overlay                             |
| `Esc`      | close overlay/log/search                               |

The most important navigation detail is that `РєЉ` and `РєЊ` should not simply move to the same column index. They should move to the nearest card by screen X-coordinate. That makes navigation feel natural when rows have different numbers of parallel jobs.

Example:

```rust
fn workflow_move_vertical(&mut self, direction: VerticalDirection) {
    let current_center_x = self.workflow_layout
        .card_center_x(self.workflow_selected_id());

    let target_row = match direction {
        VerticalDirection::Up => self.current_row().saturating_sub(1),
        VerticalDirection::Down => self.current_row() + 1,
    };

    self.workflow_selected = self.workflow_layout
        .cards_in_row(target_row)
        .min_by_key(|card| card.center_x.abs_diff(current_center_x))
        .map(|card| card.node_id.clone());
}
```

```rust
pub(crate) fn materialized_workflow_graph(
    plan: &ExternalTestPlan,
) -> ExternalWorkflowGraph;
```

This ensures the Workflow tab does not duplicate dependency logic.

# How status should be merged

A planned node and a live GitLab job need to be matched.

Use this matching order:

1. exact `job_id`
2. exact GitLab `job.name`
3. generated VTI job name, such as `vti-<subsystem>-<index>`
4. external testmap job name
5. command hash fallback

Each planned node should carry a stable ID:

```rust
workflow:vti:<plan_id>:<test_id>
workflow:proof:<lane_name>:<command_hash>
workflow:gitlab:<pipeline_id>:<job_id>
workflow:external:<plan_id>:<job_name>
```

This preserves selection across refreshes.

# Why not just reuse the current Flow Board?

The existing `FlowGraphWidget` in `src/tui/flow/widget.rs` is good for pipeline stages, but it is not the right primary abstraction for this request.

It currently does this:

```text
Commit | Admission | Impact | Pipeline | Build | Tests | Security | Release
```

That is a left-to-right pipeline map.

The requested view is this:

```text
row 0: prerequisite
row 1: parallel required tests
row 2: dependent tests/gates
row 3: release/security/final gates
```

That is a top-to-bottom dependency map.

So I would reuse concepts from `flow/model.rs` and `flow/collector.rs`, but implement a new `workflow` module with true dependency ranks.

# Example workflow scenarios

## Leaf TUI change

# Final recommended design

The best version is a new default `0:Workflow` tab backed by a real dependency graph:

```text
source data:
  GitLab live jobs
  + latest VTI test plan
  + external testmap dependencies
  + proof-lanes.toml

normalized into:
  WorkflowNode
  WorkflowEdge
  WorkflowTier
  WorkflowSnapshot

rendered as:
  dependency rows top-to-bottom
  parallel jobs side-by-side
  cards colored by waiting/running/ran/error
  inspector on the right
  keyboard-first navigation
```

This gives JeRyu a TUI view that feels purpose-built for agentic CI/CD: not just a list of jobs, but a living proof workflow where the user can see what must run, what can run together, what is blocking, and what failed.
