## Bottom line

The current API/TUI has a strong foundation, but it is still closer to a **polling observability dashboard** than a world-class **agent mission-control cockpit**. The biggest change I would make is architectural: create a dedicated, typed, real-time **TUI/API backend** that streams state, evidence, actions, blockers, and recommendations into the TUI. Right now, the TUI owns DB/Docker/GitLab clients and assembles its own view state from many places; that works for a prototype, but it limits UX, real-time insight, consistency, and actionability. The TUI docs explicitly describe the app owning database, Docker, and GitLab handles, running refresh workers, polling logs, and refreshing general state roughly every 1500 ms, with no websocket transport. ([GitHub][1])

My north star would be:

> **Every TUI screen should answer: what is happening, why is it happening, what is blocked, what is risky, and what is the safest next action?**

# 1. Current API/TUI: what is already good

## The API surface is unusually broad

The API docs describe a fairly complete control surface: CLI, TUI, webhook HTTP API, Unix-socket capability API, Git server hook, custom executor, action registry, state API, settings API, GitLab REST wrapper, release API, and secrets API. That is a strong base for a terminal mission-control UI because the core primitives already exist somewhere in the system. ([GitHub][2])

The CLI also exposes useful agent/TUI actions: job trace/explain/retry, pipeline explain/doctor/jobs/bottlenecks, cache status/gc, agent spawn/list/merge, test run/plan/select/audit/learn/cache-status, release status/watch/doctor/preflight, progress, next action, explain-blocker, and action list. ([GitHub][2])

## The TUI is already more than a toy

The TUI has nine top-level tabs: `Mission`, `Release`, `Jobs`, `Agents`, `Tests`, `Pools`, `Cache`, `Evidence`, and `Secrets`; it also supports smoke rendering and deterministic PNG capture for tabs like jobs, tests, evidence, and secrets. ([GitHub][1])

The current implementation has a useful “stable under partial data” design. Flow snapshots do not blank the board when fresh data is temporarily unavailable; instead, the previous board is retained and marked stale. That is exactly the right instinct for a live operations dashboard. ([GitHub][1])

The log UX is also pointed in the right direction: selected-job log view, follow-tail behavior, scrollback, and stale/error marking when log fetch fails. But it is still polling GitLab traces every 650 ms rather than using a streaming transport. ([GitHub][1])

## The action registry is the right idea

The action registry is meant to be the single source of truth for TUI, CLI, and capability actions. It includes action IDs, labels, risk tiers, surfaces, side-effect classes, required grants, and dry-run support. That is exactly what a world-class TUI needs for a command palette and safe action execution. ([GitHub][2])

This is a very important foundation: the TUI should not hardcode “press `r` to retry” logic forever. It should ask the action system, “what can I do to this selected entity, what risk does it carry, why is it enabled/disabled, and what will happen?”

# 2. What the current API/TUI is missing

## 2.1 A dedicated real-time TUI API

The current TUI pulls from DB, GitLab, Docker, filesystem checks, release evidence files, cache counters, and recursive directory sizes. The docs list those sources directly. ([GitHub][1])

That creates several UX problems:

* The TUI has to understand too much backend logic.
* Every tab risks drifting from CLI/API behavior.
* Polling becomes the default, not streaming.
* The UI cannot easily show causal chains like “this push caused this test plan, which caused this job, which produced this capsule, which blocks this merge.”
* Agents and humans do not get the same consistent snapshot.

**Fix:** create a TUI-specific backend-for-frontend API, even if it is local-only.

I would add:

```http
GET /v1/tui/snapshot?project_id=2&ref=main
GET /v1/tui/stream?project_id=2&ref=main
GET /v1/tui/entity/:kind/:id
GET /v1/tui/actions?entity_kind=job&entity_id=123
POST /v1/tui/actions/:action_id/dry-run
POST /v1/tui/actions/:action_id/execute
```

The TUI should subscribe to typed events instead of reconstructing state itself.

---

## 2.2 No unified “mission state” model

The current `TuiStateSnapshot` contains lots of useful fields: pools, GitLab readiness, containers, jobs, pipelines, flow, live logs, cache metrics, storage, release status, test bottlenecks, evidence, secret audit events, agent pipelines, and audit events. ([GitHub][1])

But that is still a **bag of observability fields**, not a mission model.

A world-class TUI needs a single top-level mission state:

```json
{
  "health": "degraded",
  "project": {"id": 2, "name": "dougx", "ref": "main"},
  "top_blocker": {
    "kind": "job",
    "id": 14445,
    "reason": "integration-cache failed",
    "recommended_action": "open_failure_capsule"
  },
  "next_action": {
    "id": "retry_job",
    "label": "Retry failed job",
    "risk": "low",
    "why": "failure classified as transient infrastructure"
  },
  "active_agents": 3,
  "merge_ready": 1,
  "blocked_releases": 1,
  "data_freshness_ms": 420
}
```

This would let the TUI open on a screen that immediately tells the user what matters.

---

## 2.3 The TUI is tab-centered, not entity/action-centered

The tabs are sensible, but the UX should not primarily be “go to tab 7 to look at cache.” It should be “select the thing that is blocking progress, inspect why, then execute the safest next action.”

The current Flow screen combines release watch, flow board, live jobs, and log preview. It renders only the first active pipeline, while the docs list “Flow Board renders only the first active pipeline” as a current limitation. ([GitHub][1])

**Redesign:** use a four-zone layout:

```text
┌────────────────────────────────────────────────────────────────────────────┐
│ Project dougx/main | GitLab OK | Agents 3 | Pipelines 5 | Cache 82% | 420ms │
├───────────────┬───────────────────────────────────────────┬────────────────┤
│ NAV           │ MAIN VIEW                                 │ INSPECTOR      │
│ Mission       │ Pipeline DAG / Agent timeline / Evidence  │ Why blocked    │
│ Agents        │                                           │ Risk           │
│ Pipelines     │                                           │ Next actions   │
│ Tests         │                                           │ Proof          │
│ Evidence      │                                           │ Shortcuts      │
│ Release       │                                           │                │
│ Security      │                                           │                │
│ Cache         │                                           │                │
├───────────────┴───────────────────────────────────────────┴────────────────┤
│ Event stream / logs / command palette                                      │
└────────────────────────────────────────────────────────────────────────────┘
```

The right inspector is the key. Every selected job, agent, test plan, release, cache taint, or secret event should have an inspector with “why,” “risk,” and “actions.”

---

## 2.4 The command palette needs to become the main operating surface

The action registry already knows actions, risk, surfaces, and grant requirements. ([GitHub][2])

But the TUI should go further. `Ctrl-K` should open a searchable command palette that is context-aware:

```text
Actions for job #14445 integration-cache

[Enter] Open failure capsule          read-only
[r]     Retry job                     low risk       available
[c]     Cancel job                    low risk       available
[e]     Explain blocker               read-only
[s]     Search similar failures       read-only
[q]     Quarantine cache namespace    high risk      requires confirmation
[m]     Mark as infra flake           high risk      disabled: needs reviewer grant
```

Each action should include:

```json
{
  "id": "retry_job",
  "available": true,
  "disabled_reason": null,
  "risk": "low",
  "required_grant": "job_control",
  "dry_run_supported": true,
  "preview": {
    "will_call": "gitlab.retry_job",
    "target": {"project_id": 2, "job_id": 14445},
    "expected_result": "new job attempt in same pipeline"
  },
  "undo": null
}
```

Right now `ListAllowedActions` is static-contract oriented; it lists action metadata but does not fully answer “is this action available for the selected entity right now, and why?” ([GitHub][2])

## 2.5 The API should stream logs, not poll whole traces

The docs say live logs are currently polling-based and call `job_trace(project_id, job_id)` every 650 ms; there is no websocket transport. ([GitHub][1])

That will feel slow and wasteful as soon as agents run many jobs. A world-class TUI needs cursor-based logs:

```http
GET /v1/jobs/:job_id/log/stream
GET /v1/jobs/:job_id/log?cursor=abc&limit=8192
```

Response:

```json
{
  "job_id": 14445,
  "cursor": "byte:184320",
  "chunks": [
    {
      "offset": 181200,
      "text": "...",
      "kind": "stderr",
      "diagnostics": [
        {
          "severity": "error",
          "message": "assertion failed",
          "file": "src/cache.rs",
          "line": 212
        }
      ]
    }
  ],
  "truncated": false,
  "complete": false
}
```

The TUI should render log sections, not just raw text:

```text
Compile        ✓
Unit tests     ✓
Integration    ✗  failure at src/cache.rs:212
Artifacts      ✓
```

## 2.6 The pipeline graph is not visual enough

The docs describe a Flow Board with columns inferred from job names, lane inference, and nodes painted left-to-right. ([GitHub][1])

The docs also say graph edges are not computed and `FlowGraph.edges` is empty, but the current source appears to now build coarse stage-ordering edges and compute a critical path from them. ([GitHub][1])

That is progress, but the TUI needs a real DAG, not just columns. It should show:

```text
Admission ──▶ Impact ──▶ Build ──▶ Tests ──▶ Release Gate ──▶ Canary ──▶ Prod
   ✓            ✓          ✓        ✗
                                      │
                                      └─ integration-cache #14445 failed
```

The graph should use GitLab `needs`, downstream/bridge pipelines, child pipeline relationships, release gates, and VTI-selected/skipped jobs. Where exact dependencies are unknown, the UI should label inferred edges differently from real edges.

Suggested edge styles:

```text
──▶ exact dependency
- -▶ inferred stage edge
══▶ critical path
··▶ skipped by VTI
```

## 2.7 The Agent tab should show agent sessions, not just agent-ish pipelines

The TUI snapshot includes `agent_pipelines` and recent audit events, and the current UI has an Agents tab. ([GitHub][3])

But for an agent-first product, the primary entity should be an **agent session**:

```text
agent-cache-fix-42
Goal: Fix cache epoch regression
Branch: agent/cache-fix-42
MR: !81
State: debugging failed integration job
Attempt: 3 / 6
Budget: 18m / 45m, $2.31 / $10
Trust: untrusted
Active grants: read_logs, write_agent_branch
Blocked by: integration-cache #14445
Next action: inspect capsule
Risk: medium
```

The current capability API exposes intents like `ProposePatch`, `RacePatches`, `RunTests`, `FetchCapsule`, `RequestMerge`, `ExplainBlockers`, `GetSystemSnapshot`, `ListAllowedActions`, and `PlanValidation`. ([GitHub][2])

The TUI should show those intents as a timeline:

```text
12:01  observed failure capsule #cap_9aa
12:03  proposed patch to src/cache.rs
12:04  opened MR !81
12:05  ran selected tests: unit-cache, integration-cache
12:08  integration-cache failed
12:09  generated hypothesis race: A/B/C
```

## 2.8 Plan validation is too shallow for a great Tests UI

The capability implementation for `PlanValidation` currently ignores `project_id`, checks global selector misses in the last 24 hours, and returns validity based on whether that count is zero. ([GitHub][4])

That is not enough for a world-class VTI/test TUI.

The Tests tab should show a full validation proof:

```text
Diff: abc123..def456

Changed files:
  src/cache/epoch.rs
  src/cache/taint.rs

Impacted subsystems:
  cache-core          confidence 0.97
  executor-cache      confidence 0.82

Selected tests:
  unit-cache          required
  integration-cache   required
  cache-taint-smoke   sentinel

Skipped tests:
  full-e2e            skipped, confidence 0.93
  release-prod        skipped, no release-impacting path

Risk:
  selector misses last 7d: 0
  unknown paths: 0
  global invalidators touched: no
  cache taints active: 0

Decision:
  selected plan is valid
```

The API should support this with:

```http
GET /v1/tests/plan?project_id=2&base=main&head=agent/cache-fix
GET /v1/tests/plan/:id/proof
GET /v1/tests/selector-misses?project_id=2&ref=main
```

## 2.9 Merge/release proof needs to be first-class in the TUI

`RequestMerge` currently returns a merge-gate proof, but the docs call the current proof conservative/advisory for selector and cache state. ([GitHub][2])

The source also shows that the capability merge path uses simplified inputs like `successful_jobs: 1`, `pending_jobs: 0`, `failed_jobs: 0`, and a policy with evidence requirements disabled. ([GitHub][4])

That means the TUI should not show “merge allowed” as a simple green check. It should show a **proof checklist**:

```text
Merge proof for !81

Head SHA bound to proof        ✓ def456
Required jobs passed           ✓ 12 / 12
Pending jobs                   ✓ 0
Recent failures                ✓ none
Selector misses                ✓ 0
Cache taints                   ✓ 0
Secrets touched                ✓ no
Protected paths touched        ✓ no
Agent grant matched branch     ✓ yes
Human approval required        ✗ missing approval from release owner

Decision: ESCALATE
Next action: request release-owner approval
```

The API should expose:

```http
GET /v1/merge/:mr_iid/proof?project_id=2
GET /v1/release/:version/proof
GET /v1/release/current/gates
```

## 2.10 The Audit/Evidence experience needs to become a searchable ledger

The docs explicitly list “Audit tab is a placeholder” and “job inspector renderer exists but is not wired into the active layout” as current limitations. ([GitHub][1])

This is one of the biggest TUI gaps. The Evidence tab should become the most important debugging screen.

It should support:

```text
Filter: [job:14445] [kind:failure_capsule] [agent:agent-cache-fix] [risk:high]

12:03:15  agent.intent.created         FetchCapsule
12:04:02  agent.patch.proposed         branch agent/cache-fix
12:05:44  admission.allow              refs/heads/agent/cache-fix
```

Selecting any event should show provenance:

```text

SHA: def456
```

# 3. API redesign from the ground up

## 3.1 Create a local TUI service layer

Right now the TUI is effectively a client plus backend combined. I would split it:

```text
jeryu serve
  ├─ event ingestion
  ├─ state database
  ├─ GitLab/Docker/cache/release clients
  ├─ action executor
  ├─ evidence engine
  └─ TUI API

jeryu tui
  └─ thin terminal client
```

This unlocks multiple simultaneous TUIs, remote dashboards, replay, web UI later, and agent clients that see the same truth.

## 3.2 Standardize every response envelope

Every API response should include freshness and action context:

```json
{
  "schema_version": "tui.v1",
  "request_id": "req_123",
  "generated_at": "2026-04-27T18:01:22Z",
  "source_lag_ms": 420,
  "stale": false,
  "warnings": [],
  "data": {},
  "actions": [],
  "links": {}
}
```

The `source_lag_ms` and `stale` fields are critical. Users should always know whether they are seeing live state or retained state.

## 3.3 Add typed streaming events

```http
GET /v1/tui/stream
```

Event types:

```text
system.health.updated
pipeline.created
pipeline.updated
pipeline.graph.updated
job.updated
job.log.chunk
job.failure_capsule.created
test.plan.created
test.selector_miss.created
agent.session.created
agent.intent.started
agent.intent.finished
agent.grant.created
admission.decision.created
cache.taint.created
cache.gc.plan.created
release.gate.updated
release.promoted
secret.audit.created
next_action.updated
```

The TUI can then update just the cells that changed, instead of rehydrating broad snapshots.

## 3.4 Add entity APIs

```http
GET /v1/entities/job/:id
GET /v1/entities/pipeline/:id
GET /v1/entities/agent/:id
GET /v1/entities/release/:version
GET /v1/entities/cache-taint/:id
GET /v1/entities/evidence/:id
```

Each entity should return:

```json
{
  "summary": {},
  "timeline": [],
  "blockers": [],
  "evidence": [],
  "related_entities": [],
  "available_actions": []
}
```

This maps perfectly to the right-side inspector.

## 3.5 Add contextual action APIs

The static action registry is good, but the TUI needs dynamic action availability. ([GitHub][2])

```http
GET /v1/actions/context?entity_kind=job&entity_id=14445
POST /v1/actions/retry_job/dry-run
POST /v1/actions/retry_job/execute
```

A contextual action response should include:

```json
{
  "id": "retry_job",
  "label": "Retry job",
  "risk": "low",
  "available": true,
  "disabled_reason": null,
  "requires_confirmation": false,
  "required_grant": "job_control",
  "dry_run": {
    "supported": true,
    "summary": "Will retry failed job #14445 in project 2"
  },
  "result_shape": {
    "new_job_id": "number",
    "pipeline_id": "number"
  }
}
```

## 3.6 Add a TUI graph API

```http
GET /v1/pipelines/:id/graph
```

Response:

```json
{
  "pipeline_id": 9821,
  "nodes": [
    {
      "id": "job:14445",
      "label": "integration-cache",
      "status": "failed",
      "lane": "Integration",
      "column": "Tests",
      "required": true,
      "critical_path": true,
      "duration_ms": 184000,
      "eta_ms": null,
      "actions": ["open_logs", "fetch_capsule", "retry_job"]
    }
  ],
  "edges": [
    {
      "from": "job:build",
      "to": "job:integration-cache",
      "kind": "gitlab_needs"
    }
  ],
  "critical_path": ["job:build", "job:integration-cache"],
  "blockers": ["job:14445"]
}
```

The TUI should not infer the graph by job-name heuristics unless no better data exists.

## 3.7 Add an evidence API designed for humans and agents

```http
GET /v1/evidence?project_id=2&ref=main&limit=100
GET /v1/jobs/:id/evidence
GET /v1/evidence/:id/repro
GET /v1/evidence/:id/related
```

Each capsule should include:

```json
{
  "id": "cap_123",
  "job_id": 14445,
  "classification": "regression",
  "failure_kind": "test_failure",
  "summary": "cache epoch mismatch",
  "repro_command": "cargo test cache_epoch",
  "log_slice": {},
  "suspected_files": ["src/cache/epoch.rs"],
  "retry_advice": "do_not_retry",
  "related_failures": []
}
```

# 4. TUI redesign: world-class UX

## 4.1 Mission tab: make it the command center

The Mission tab should not be a generic status page. It should be the “what matters now?” page.

```text
JeRyu Mission Control
Project: dougx / main                         Data: live, 420ms lag

System: DEGRADED          GitLab: OK          Runners: 6/8 warm
Agents: 3 active          Pipelines: 5        Cache: 82% hit
Release: blocked          Merge-ready: 1      Secrets: clean

Top blocker:
  integration-cache #14445 failed in pipeline #9821
  Classification: regression
  Impact: blocks release ci-def456
  Next action: open failure capsule

Recommended actions:
  1. Open capsule for job #14445                         Enter
  2. Run VTI proof for branch agent/cache-fix             v
  3. Request human approval for MR !81                    a
```

This screen should be powered by `/v1/tui/snapshot` and `/v1/next-action`.

## 4.2 Pipelines tab: turn Flow Board into a true DAG cockpit

Current Flow Board is the most important screen, but it needs richer visual structure. The docs say it renders columns and nodes, with progress and ETA heuristics. ([GitHub][1])

New design:

```text
Pipeline #9821  main@def456  74%  ETA 3m12s  Critical path: integration-cache

Admission     Impact       Build        Tests             Release       Canary
   ✓            ✓            ✓       ┌─ unit-cache ✓
                                     ├─ lint ✓
                                     ├─ integration-cache ✗  [CRIT]
                                     └─ e2e skipped by VTI

Blocker: integration-cache failed
Why: regression, not infra flake
Action: open capsule / retry disabled / create agent task
```

Features:

* Multiple active pipelines, selectable with `[` and `]`.
* Real DAG edges.
* Critical path highlighted.
* Blocker node pulsing or bolded.
* Skipped jobs shown dimmed with reason.
* Downstream/child pipelines folded into expandable groups.
* Agent-owned jobs marked with an agent glyph.
* Failed jobs show capsule availability.
* Long-running jobs show “over expected p95.”

## 4.3 Agent tab: show agent intent, authority, and progress

New design:

```text
Agents

agent-cache-fix-42        RUNNING      trust: untrusted      budget: 18m/45m
Goal: Fix cache epoch regression
Branch: agent/cache-fix-42     MR: !81
Current: reading capsule #cap_123
Next: propose patch
Risk: medium
Grants: read_logs ✓ write_branch ✓ merge ✗

Timeline:
  ✓ Spawned task
  ✓ Created branch
  ✓ Opened MR
  ✓ Ran selected tests
  ✗ integration-cache failed
  → Inspecting failure capsule
```

Right inspector:

```text
Why this agent is blocked:
  Waiting on failed job evidence.

Available actions:
  Open capsule
  Pause agent
  Revoke write grant
  Approve next patch
  Escalate to human
```

This should be backed by real `agent_session` objects, not only pipeline records.

## 4.4 Tests tab: make VTI explain itself

The current Tests tab focuses on bottlenecks and history. The API docs expose much richer test controls: plan, select, select-external, audit, learn, cache-status, and explain-plan. ([GitHub][2])

The TUI should show test selection as a proof:

```text
Validation Plan for agent/cache-fix-42

src/cache/taint.rs         executor-cache 0.82      integration-cache

misses last 24h: 0

unknown paths: 0
```

# 5. Better graphics, colors, and real-time feel

## Color system

Current status colors are simple: green for success, blue for running, red for failed, yellow for pending/created, dark gray for canceled, and cyan for active focus. ([GitHub][1])

I would replace that with a semantic palette:

```text
Healthy / success      green
Running / live         cyan or blue
Waiting / pending      amber
Blocked                magenta
Failed                 red
Danger / production    bold red
Skipped / omitted      dim gray
Stale data             gray + clock glyph
Agent-owned            violet or cyan accent
Human-required         amber + person glyph
Security-sensitive     red/orange shield glyph
```

Add themes:

```text
theme dark
theme dark-high-contrast
theme light
theme colorblind
theme no-nerd-font
```

Do not rely on color alone. Use glyphs and labels:

```text
✓ success
● running
… pending
✗ failed
⚠ blocked
⏱ stale
🔒 protected
🤖 agent
```

## Graphics

Use terminal-safe visuals first:

* Gauges.
* Sparklines.
* Timelines.
* DAG edges.
* Heatmaps.
* Compact histograms.
* Unicode Braille charts.
* Optional Kitty/Sixel image protocol for richer graphs where supported.

Examples:

```text

```

## Real-time feedback

Instead of redrawing everything equally, the TUI should visually indicate changed cells:

```text
job #14445  running → failed   flash red for 1s
cache hit   81% → 82%          pulse green
agent       planning → patching
```

Add a small freshness ribbon:

```text
Data: live 420ms | stream connected | last event 0.3s ago
```

If the stream drops:

```text
Data: stale 14s | reconnecting | showing retained state
```

This builds trust.

# 7. Highest-impact TUI changes

## P0 TUI work

1. **Replace raw polling UX with event-stream UX.** Keep polling fallback, but prefer `/v1/tui/stream`.

2. **Add right-side inspector everywhere.** Every selected entity gets blockers, evidence, risk, and actions.

3. **Make Mission tab actionable.** Show top blocker, next best action, merge-ready work, blocked releases, and stale data.

4. **Render real pipeline DAGs.** Show edges, critical path, blockers, skipped jobs, downstreams, and selected node details.

5. **Turn Agents into session view.** Show goal, branch, MR, state, budget, grants, timeline, and next intended action.

6. **Make Evidence/Audit real.** Searchable event ledger, capsule inspector, provenance, filters.

7. **Upgrade Tests to validation proof.** Show changed files → impacted subsystems → selected/skipped tests → confidence → decision.

8. **Make command palette context-aware.** Dynamic actions, disabled reasons, dry-run preview, risk confirmation.

9. **Add project/ref selector.** The current TUI is too tied to default release project/ref behavior; users need to switch project, ref, branch, MR, agent, or release context.

10. **Add themes and accessibility.** High contrast, colorblind, no-glyph, and no-Nerd-Font modes.

# Final recommendation

The current API/TUI has good raw pieces: broad CLI/API coverage, a Unix-socket capability API, webhook ingestion, action registry, Flow Board, live jobs, log preview, release state, cache/test/evidence/secrets surfaces, and anti-blanking snapshot behavior. ([GitHub][2])

The missing world-class layer is a **typed, real-time, entity/action/proof API** that feeds a TUI designed around decisions instead of tabs. Build the TUI around five questions:

1. **What is happening?**
2. **Why is it happening?**
3. **What is blocked?**
4. **What is risky?**
5. **What should happen next?**

That shift would turn JeRyu from a capable terminal dashboard into an actual agent-first command center.
