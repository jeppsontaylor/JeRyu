I focused only on the **current API** and **current TUI**. My strongest conclusion: the TUI has good bones, but the platform needs a **dedicated, versioned TUI/agent API** and the UI should be redesigned around **causality, trust, blockers, and next safe action**, not around loosely connected status panels.

## Executive diagnosis

The current API surface is broad: CLI, TUI, webhook server, Unix-socket capability API, Git server hook, custom executor, action registry, state API, settings, GitLab REST wrapper, release automation, and secrets automation are all documented as public-ish surfaces. That breadth is powerful, but it also means the TUI is currently stitching together too much state from too many places instead of consuming one coherent “operator view” API. The API docs call out those surfaces directly, along with fixed defaults like GitLab ports, default release project, and default release repo root. ([GitHub][1])

The current TUI is more advanced than the older TUI docs suggest. The docs say the TUI has nine capture targets — Mission, Release, Jobs, Agents, Tests, Pools, Cache, Evidence, Secrets — and the source confirms the active tab enum maps keys 1–9 to those tabs. ([GitHub][2]) But the docs also still describe older five-tab labels like Flow, Storage, Cache, Audit, and Tests, and they list limitations such as an Audit placeholder, inactive pool/pipeline renderers, and job inspector not wired. ([GitHub][2]) Current source contradicts some of that: the Jobs tab now renders a pipeline navigation column, release banner, flow board, jobs/logs, and a right-side job inspector. ([GitHub][3])

That doc/source drift is not cosmetic. It is a product problem. For a world-class agent-first TUI, the API, docs, command palette, screenshots, and source must all agree on the same mental model.

# What the current TUI/API already does well

## 1. It has the right conceptual surfaces

The TUI is not just “jobs and logs.” It already has Mission, Release, Jobs, Agents, Tests, Pools, Cache, Evidence, and Secrets tabs in source, which is close to the right top-level information architecture for agentic coding operations. ([GitHub][4])

That matters because an agent-first Git system needs to answer more than “did CI pass?” It needs to answer:

* What is blocked?
* Why is it blocked?
* What agent is acting?
* What proof exists?
* What cache/test/release/security state is trustworthy?
* What should happen next?

JeRyu is already gesturing toward that model.

## 2. The TUI has a real mission-control structure

The current source has a Mission tab that draws a two-by-two layout and includes system health details like GitLab, containers, pools, proxy, and registry. ([GitHub][3]) The Release tab has a gate matrix and side panel, with gates for main branch, release identity, release pipeline, canary health, E2E, and production promotion. ([GitHub][3])

That is the right direction. Most CI dashboards are passive. This one is trying to become an operational cockpit.

## 3. The action registry is a strong primitive

The action registry exposes entries with IDs, labels, key hints, risk tier, surfaces, dry-run flag, and description. It also models side-effect classes and required grants, and contains actions such as open logs, retry job, delete agent branch, pause flow, explain blockers, get system snapshot, propose patch, race patches, request merge, plan validation, run tests, and next action. ([GitHub][5])

This is very promising. The missing piece is that the registry should become an executable, previewable, grant-aware action framework, not just a menu/catalog.

## 4. There is already a flow model, not just a flat job list

The flow model has `FlowSnapshot`, `PipelineFlow`, `FlowGraph`, columns, nodes, edges, current blocker, critical path, ETA, and progress percentage fields. ([GitHub][6]) The builder classifies jobs into columns by name, builds graph edges between stages, and computes critical path. ([GitHub][7])

That said, the graph is still inferred mostly from job names and heuristics, not from a first-class causal validation graph. That is one of the most important redesign opportunities.

## 5. The API has agent-facing capability concepts

The capability API supports typed intents such as `ProposePatch`, `RacePatches`, `RunTests`, `FetchCapsule`, `RequestMerge`, `ExplainBlockers`, `GetSystemSnapshot`, `ListAllowedActions`, and `PlanValidation`. It uses a Unix socket and JSON payloads. ([GitHub][1])

This is exactly the kind of API shape an agent-first Git system needs. The current version is still too small and too one-shot, but the existence of typed intents is a major plus.

# The biggest missing thing: a dedicated TUI Gateway API

Right now, the TUI appears to assemble its view from many sources: GitLab job traces, state tables, filesystem/`df`, cache TCP checks, release status, test bottlenecks, recent evidence, secret audit events, agent pipelines, and event records. The snapshot loop in source hydrates all of these directly and sleeps roughly 1500 ms between general syncs. ([GitHub][4]) Logs are polled separately from GitLab job traces every 650 ms. ([GitHub][2])

That is fine for a prototype, but not for a world-class TUI.

The TUI should not be a pile of ad hoc polling loops. It should consume a **single coherent operator API**:

```text
/v1/tui/snapshot
/v1/tui/events/stream
/v1/tui/flow/:pipeline_id
/v1/tui/agents
/v1/tui/actions
/v1/tui/actions/:id/preview
/v1/tui/actions/:id/execute
/v1/tui/logs/:job_id/stream
/v1/tui/evidence/:id
/v1/tui/releases/current
/v1/tui/tests/impact
/v1/tui/cache/health
/v1/tui/security/state
```

The TUI API should own:

* schema versioning
* snapshot versioning
* event sequence numbers
* stale-state markers
* causal links between records
* structured action previews
* permission/grant requirements
* optimistic updates
* reconnect/backfill behavior
* compact terminal-friendly summaries

This one change would improve UX, speed, reliability, and future extensibility more than any color/theme change.

# Current API gaps that directly hurt the TUI

## 1. The capability API is too one-shot

The capability server reads one JSON payload into a 65,536-byte buffer, deserializes it, handles one intent, and writes one JSON response. ([GitHub][8]) The docs also describe the capability API as a Unix-domain socket JSON interface with one payload/response and a 64 KiB buffer. ([GitHub][1])

That is too limited for a world-class TUI and agent console. It needs:

* sessions
* streaming responses
* pagination
* cancellation
* progress events
* artifact references instead of large inline payloads
* idempotency keys
* action leases
* structured errors
* schema versions
* correlation IDs
* dry-run previews
* grant negotiation

A better shape:

```json
{
  "schema_version": "tui.capability.v2",
  "request_id": "req_01",
  "session_id": "tui_abc",
  "actor": {
    "kind": "human",
    "id": "local-user"
  },
  "intent": "run_tests",
  "subject": {
    "repo": "myrepo",
    "ref": "agent/fix-auth",
    "sha": "abc123"
  },
  "options": {
    "scope": "impacted",
    "stream": true,
    "dry_run": false
  },
  "idempotency_key": "run-tests-abc123-impacted"
}
```

Then the TUI can show real progress instead of waiting for a single response.

## 2. `GetSystemSnapshot` is too thin

`GetSystemSnapshot` currently returns pool summary, cache hit/miss/object counts and hit ratio, selector misses in the last hour, recent failures, and generated timestamp. ([GitHub][8]) That is useful, but it is not enough to drive a rich Mission tab.

A world-class snapshot should include:

```json
{
  "health": {
    "overall": "degraded",
    "gitlab": "ok",
    "database": "ok",
    "event_bus": "ok",
    "runner_capacity": "constrained",
    "cache": "tainted",
    "secrets": "ok"
  },
  "top_blocker": {
    "kind": "failed_test",
    "title": "auth_refresh integration shard failing",
    "subject": "pipeline/123/job/456",
    "impact": "blocks auto-merge for !42",
    "next_action": "open_failure_capsule"
  },
  "active_agents": [],
  "active_pipelines": [],
  "release": {},
  "risk": {},
  "next_actions": []
}
```

The Mission tab should never force a user to inspect five panels to understand the state of the system.

## 3. `RequestMerge` is not proof-grade yet

The current `RequestMerge` implementation checks recent selector misses and active cache taints, then calls the merge gate with hardcoded successful/pending/failed job counts and conservative policy flags. ([GitHub][8]) The API docs also say `RequestMerge` currently returns a proof but the merge proof is conservative/advisory and is intended to move into the shared gate later. ([GitHub][1])

For the TUI, that means the “Request Merge” action should be displayed as **advisory**, not authoritative. A world-class redesign should make merge readiness a first-class endpoint:

```text
GET /v1/proofs/merge-readiness?project_id=2&mr=42
```

It should return:

* exact MR head SHA
* base SHA
* pipeline IDs
* selected tests
* skipped tests
* cache artifacts consumed
* taints
* selector confidence
* release impact
* known failures
* required approvals
* final verdict
* exact blockers
* remediation commands

The UI should show:

```text
MERGE BLOCKED
Reason 1: integration/auth_refresh failed on exact SHA abc123.
Reason 2: VTI skipped 14 tests with selector confidence 0.71.
Reason 3: cache artifact build-layer-9 consumed from untrusted namespace.

Recommended next action:
Run expanded validation plan on auth subsystem.
```

## 4. `PlanValidation` is too shallow

The current `PlanValidation` implementation ignores `project_id`, treats selector misses as global, and mainly determines validity from whether there were selector misses in the last 24 hours. ([GitHub][8]) That is not enough to drive a great Tests/Impact tab.

A better validation plan API should return:

* changed files
* impacted subsystems
* selected tests
* skipped tests
* confidence
* historical selector misses
* flaky-test risk
* estimated runtime
* cost of full validation
* recommended widening
* why each test was included or skipped
* whether the plan is sufficient for merge/release

The TUI should let a user press one key on any MR or pipeline and see: **“What proof do I still need?”**

## 5. `RacePatches` needs a lifecycle API

`RacePatches` currently launches hypothesis branches and says pipeline polling is needed to determine the winner, while losing branches need manual cleanup or a future `PollRaceResult`. ([GitHub][8])

That is a perfect TUI opportunity, but the API needs a real race object:

```text
POST /v1/races
GET  /v1/races/:id
GET  /v1/races/:id/events
POST /v1/races/:id/cancel
POST /v1/races/:id/promote-winner
POST /v1/races/:id/cleanup-losers
```

The Agents tab could then show:

```text
Hypothesis Race: fix-auth-refresh
A: mock clock isolation       running    confidence 61%
B: retry token refresh lock   failed     root cause: deadlock
C: widen session TTL          passed     risk: medium

Recommended: promote C, run expanded auth validation.
```

## 6. The action registry needs execution semantics

The registry currently has metadata fields, and side-effect class / required grant are derived from action IDs. ([GitHub][5]) That is a good start, but the TUI needs each action to expose:

* preconditions
* subject type
* preview renderer
* required grant
* required confirmation phrase
* dry-run command
* execute command
* rollback/recovery command
* expected events
* timeout
* destructive risk
* audit entry shape

Example:

```json
{
  "id": "delete_agent_branch",
  "label": "Delete agent branch",
  "risk": "destructive",
  "subject_types": ["agent_branch"],
  "preconditions": [
    "branch_not_main",
    "no_active_pipeline",
    "no_open_merge_request"
  ],
  "preview_endpoint": "/v1/actions/delete_agent_branch/preview",
  "execute_endpoint": "/v1/actions/delete_agent_branch/execute",
  "requires_confirmation": "type branch name",
  "rollback": {
    "available": false,
    "message": "Branch deletion cannot be automatically undone."
  }
}
```

The current action palette should become a safe command system, not just a list of things the user can do.

# Current TUI gaps

## 1. The main UX problem is causality, not color

The Flow model already has columns, nodes, edges, current blocker, critical path, ETA, and progress. ([GitHub][6]) But the graph is still largely built by classifying job names into lanes, and ETA uses fallback durations by lane type. ([GitHub][7])

A world-class TUI needs a **causal graph**:

```text
Agent goal
  → patch transaction
    → branch admission
      → impact analysis
        → validation plan
          → pipeline jobs
            → failure capsule
              → retry decision
                → merge proof
                  → release gate
                    → canary
                      → production
```

Every node should answer:

* What happened?
* Why did it happen?
* What does it prove?
* What depends on it?
* Who/what caused it?
* What is the next safe action?

The TUI should not just show that a test failed. It should show why that test was selected, what changed that likely caused it, what proof is missing, and what action is safest.

## 2. The Mission tab should become the primary screen

Right now, the source default tab is Jobs, and Mission is one of the tabs. ([GitHub][4]) For a world-class UX, Mission should be the default because users and agents first need to know **state, risk, blockers, and next action**.

A redesigned Mission screen:

```text
┌ JeRyu Mission Control ──────────────────────────────────────────────────┐
│ Overall: DEGRADED   GitLab: OK   DB: OK   Runners: HOT 3/5   Cache: TAINT  │
│ Release: BLOCKED    Agents: 2 active   Merge Risk: MEDIUM   Events: LIVE   │
└────────────────────────────────────────────────────────────────────────────┘

┌ Next Best Action ──────────────────────────────────────────────────────────┐
│ Run expanded auth validation for MR !42.                                  │
│ Why: VTI confidence is 0.71 and auth/session.rs lacks owner mapping.       │
│ Cost: ~7m   Risk: low   Required grant: test.run                          │
│ [Enter] Preview   [x] Execute   [?] Explain   [e] Evidence                 │
└────────────────────────────────────────────────────────────────────────────┘

┌ Top Blockers ───────────────┐ ┌ Active Agents ─────────────────────────────┐
│ 1. auth_refresh_test failed │ │ fix-auth-7   testing hypothesis B   72%    │
│ 2. cache taint: build-layer │ │ docs-3       waiting for admission   40%   │
│ 3. release canary missing   │ │                                                │
└─────────────────────────────┘ └────────────────────────────────────────────┘

┌ Live Causal Flow ──────────────────────────────────────────────────────────┐
│ Admission → Impact → Build ✓ → Unit ✓ → Integration ✗ → Merge Proof blocked│
└────────────────────────────────────────────────────────────────────────────┘
```

The user should be able to understand the entire system in five seconds.

## 3. The Jobs tab should be renamed to Flow or Workflows

The current Jobs tab actually renders pipeline nav, release banner, flow board, jobs/logs, and inspector. ([GitHub][3]) That is more than “Jobs.” Rename it to **Flow** or **Workflows**.

Suggested tabs:

```text
1 Mission
2 Flow
3 Agents
4 Evidence
5 Tests
6 Release
7 Capacity
8 Cache
9 Security
0 Audit
```

Why this is better:

* “Flow” is the causal execution graph.
* “Evidence” is where failures/proofs live.
* “Capacity” combines pools/runners/queues.
* “Security” combines secrets, grants, taints, policy.
* “Audit” deserves its own tab instead of being hidden inside Evidence.

## 4. Evidence should become a first-class proof browser

The current state snapshot has recent evidence and recent audit event fields, and Evidence has a capsule/audit-ledger view mode. ([GitHub][4]) The current UI also has an event console that displays recent evidence capsule events. ([GitHub][3])

That should become one of the most important screens.

The Evidence tab should show:

```text
Failure Capsules
- job#123 auth_refresh_test
  kind: test_failure
  confidence: 0.86
  root cause: token refresh race after clock mock
  repro: cargo test auth_refresh -- --nocapture
  suggested action: run hypothesis race with lock-based fix

Proof Bundles
- merge-readiness !42
  exact sha: abc123
  verdict: blocked
  missing: integration shard 3, cache taint review

Audit Events
- capability.grant.created
- agent.patch.proposed
- branch.admission.accepted
- validation.plan.generated
```

A world-class agent-first system must make proof as visible as logs.

## 5. The Agents tab needs agent runs, not just agent pipelines

The current Agents tab shows agent pipelines from `app.state.agent_pipelines` and a capability call history filtered from audit events. ([GitHub][3]) That is useful, but it is not yet an agent operations screen.

Agents need durable run objects:

```text
Agent Run
- goal
- current plan
- current hypothesis
- current branch/MR
- current action
- budget used
- confidence
- blockers
- permissions/grants
- evidence collected
- pending next action
```

A redesigned Agents tab:

```text
┌ Agents ────────────────────────────────────────────────────────────────────┐
│ fix-auth-7       testing hypothesis B       confidence 72%      34m/2h     │
│ release-guard-2  waiting on canary metrics  confidence 90%      8m/30m     │
│ cleanup-1        idle                       confidence --       --         │
└────────────────────────────────────────────────────────────────────────────┘

┌ Selected Agent: fix-auth-7 ────────────────────────────────────────────────┐
│ Goal: Fix refresh-token race in auth integration tests.                    │
│ Plan: reproduce → isolate clock mock → patch session lock → run auth tests │
│ Current action: running integration/auth_refresh_test                      │
│ Branch: agent/fix-auth-20260427                                           │
│ MR: !42                                                                   │
│ Grants: branch.write, tests.run, evidence.read                             │
│ Blocker: integration shard failed once, retry classified as useful         │
└────────────────────────────────────────────────────────────────────────────┘
```

This is where a human should supervise autonomous coding.

## 6. Logs need structure, not just highlighting

The current TUI polls logs, keeps a capped tail, highlights ANSI/plain text, and preserves log visibility. ([GitHub][2]) That is necessary, but not sufficient.

Logs should be transformed into structured failure cards:

```text
Failure detected
File: src/auth/session.rs:184
Test: auth_refresh_handles_expired_session
Assertion: expected token version 2, got 1
Likely cause: refresh path did not acquire session lock
Similar failures: 3 in last 14 days
Repro: cargo test auth_refresh_handles_expired_session -- --nocapture
Suggested next action: open capsule / run focused retry / ask agent to patch
```

The raw log should still be available, but the default view should show the useful part.

## 7. Realtime insight needs event streaming, not polling

The docs explicitly say logs are polled and there is no websocket transport in the current implementation. ([GitHub][2]) The source also shows snapshot loops and log polling workers. ([GitHub][4])

World-class realtime behavior should use:

* append-only event stream
* event sequence numbers
* reconnect from last seen sequence
* log chunk offsets
* snapshot + patch model
* local reducer inside the TUI
* stale-state detection per panel
* backpressure handling
* replay/time-travel mode

The TUI should be able to say:

```text
LIVE seq=184923
Last event: validation.job.failed job#123 2s ago
Panel state: fresh
Log stream: offset 482391
```

No more blind refresh loops.

# Ground-up redesign: TUI UX/UI

## Overall layout

Use one consistent layout everywhere:

```text
┌ Header: health pills, active repo/ref, live/stale, risk, command hint ┐
├ Left: nav / list / filters ┬ Center: main graph/table/cards ┬ Right: inspector ┤
├ Bottom: event timeline / selected log / next actions                         ┤
└ Footer: shortcuts, selected subject, mode, connection state                   ┘
```

The current Jobs tab already resembles this with pipeline nav, flow board, jobs/logs, and inspector. ([GitHub][3]) Standardize that across all tabs.

## Color system

Use color as semantic language, not decoration.

Recommended palette:

```text
Success      green       completed, proven, safe
Running      cyan/blue   active work
Queued       amber       waiting, pending capacity
Failed       red         failed validation
Blocked      magenta     cannot proceed
Stale        gray        old data or disconnected source
Security     purple      secrets, grants, policy, taint
Agent        violet      autonomous actor
Human        teal        human action/approval
Production   bright red  prod-impacting action
Cache        blue        cache hit/reuse/provenance
```

Also add:

* high-contrast theme
* monochrome theme
* colorblind-safe theme
* icon/shape redundancy so status is not color-only
* stale data watermark
* per-panel freshness indicator

## Graphics that would make the TUI feel world-class

Use terminal-native visualizations:

* DAG connectors for flow
* swimlanes for validation phases
* sparklines for runner queue, cache hit rate, test duration, failures over time
* stacked bars for runner utilization
* mini heatmaps for test flakiness
* radial/gauge-like text widgets for release readiness
* tree views for evidence/proof bundles
* timeline view for agent actions
* diff-like before/after blocks for patch proposals
* warning badges for taints, stale data, missing proof

Example Flow screen:

```text
Admission ✓ ─ Impact ✓ ─ Build ✓ ─ Unit ✓ ─ Integration ✗ ─ Merge Proof BLOCKED
                                      │
                                      └ Security ✓ ─ Package ✓
```

Selected node inspector:

```text
Integration ✗
Job: auth_refresh_test
Blocks: merge proof, release candidate
Evidence: capsule#883
Likely cause: session refresh race
Next action: run focused retry or start patch hypothesis
```

## Interaction redesign

Current keybindings include tab numbers and actions like retry/delete/pause, and the command palette exists in source. ([GitHub][2]) I would make the interaction model more explicit:

```text
/        search/filter current screen
:        command palette
?        explain selected item
.        next recommended action
Enter    open/drill down
Space    select/pin
Tab      move between panes
Esc      back/close
p        preview action
x        execute approved action
r        refresh/reconnect
f        follow selected job/agent
t        time travel / event replay
```

Important change: do not overload single-letter destructive commands. A destructive action should always go through preview and confirmation.

# Redesigned tabs

## 1. Mission

Purpose: “Tell me what matters now.”

# “World-class” features I would add specifically for the TUI

## 1. Next Best Action engine

Every screen should have a recommended action:

```text
Next best action:
Run expanded validation because selector confidence is low.
```

Actions should be ranked by:

* safety
* expected value
* cost
* required grant
* reversibility
* production risk
* confidence

The action registry already has risk tiers and surfaces. ([GitHub][5]) Build on that.

## 2. Explain-anything key

Press `?` on any item:

```text
Why is this blocked?
Why was this test selected?
Why is this cache tainted?
Why is this agent paused?
Why is merge unsafe?
Why is release blocked?
```

This is crucial for agent-first UX. The user should not need to understand every subsystem.

## 3. Preview before every risky action

Before retry, delete, merge, promote, pause, race, patch, or run tests, show:

* what will happen
* what will not happen
* what it will cost
* what proof it will generate
* what grant it needs
* whether it can be undone
* what events to expect

## 4. Time-travel / replay mode

Because the system already records events and recent audit state, the TUI should let users replay:

```text
t: enter replay mode
←/→: move event by event
shift ←/→: jump 1 minute
```

This would be incredible for debugging agent behavior.

## 5. Pin and compare

Let users pin:

* two agents
* two jobs
* two pipelines
* two validation plans
* two proof bundles
* two failure capsules

Then compare side-by-side.

Especially useful for hypothesis racing.

## 6. Natural-language filter

In command palette:

```text
show failed auth jobs from last hour
show actions that require production grant
show cache taints affecting this MR
show agents touching src/auth
show why release is blocked
```

The API can map this to structured filters. It does not need an LLM to be useful.

## 7. Screenshot/report mode

The TUI already supports capture targets. ([GitHub][2]) Expand this into report generation:

```bash
jeryu tui --report mission --format markdown
jeryu tui --report release --format json
jeryu tui --report evidence --pipeline 123
```

This is useful for humans, agents, and incident review.

# Fast API changes that would immediately improve the TUI

1. **Add `/v1/tui/snapshot` or `jeryu tui snapshot --json`.**
   Stop making the TUI compose its first paint from scattered calls.

2. **Add event sequence numbers everywhere.**
   Every state mutation should have `event_seq`, `timestamp`, `actor`, `subject`, `caused_by`, and `proof_id`.

3. **Add action preview and execute endpoints.**
   The command palette becomes safe, inspectable, and automatable.

4. **Upgrade capability responses.**
   Replace generic `{ success, message, data }` with typed result objects, structured errors, progress handles, and event stream IDs. Current capability responses are generic. ([GitHub][8])

5. **Make merge readiness proof-bound.**
   Do not allow the TUI to imply merge safety from advisory or hardcoded gate inputs. Current `RequestMerge` is not enough. ([GitHub][8])

6. **Make validation planning project/ref/SHA-specific.**
   Current `PlanValidation` behavior is too global. ([GitHub][8])

7. **Create an agent-run model.**
   Agent pipelines are not enough for agent supervision. ([GitHub][3])

8. **Expose structured failure capsules as first-class TUI objects.**
   The TUI should not parse logs for everything.

9. **Expose cache trust/taint lineage.**
   Cache is only useful for autonomous coding if the UI can show whether reuse is safe.

10. **Fix API/docs/source drift.**
    The docs should reflect the current nine-tab source model, current graph behavior, current inspector wiring, and actual active/inactive screens.

# Fast TUI changes that would immediately improve UX

1. **Make Mission the default tab.**
   Jobs are implementation detail; mission state is what the user needs first.

2. **Rename Jobs to Flow.**
   The current Jobs tab already contains flow, release, logs, jobs, and inspector. ([GitHub][3])

3. **Add a persistent “Next Action” card.**
   Every tab should answer, “What should I do now?”

4. **Add freshness badges to every panel.**
   The current TUI has stale handling in places, but freshness should be visible everywhere. ([GitHub][2])

5. **Move raw logs behind structured failure summaries.**
   Raw logs are secondary; root cause, repro, and action are primary.

6. **Add a right-side inspector everywhere.**
   Selecting any row/node/card should populate the inspector with details, evidence, actions, and risk.

7. **Make the command palette contextual.**
   If a failed job is selected, top actions should be explain, open capsule, retry, run focused validation, ask agent to patch.

8. **Add search/filter globally.**
   `/` should filter current screen; `Ctrl-/` or similar should search across jobs, agents, evidence, tests, and audit.

9. **Add proper empty states.**
   Empty should never just look blank. It should say “No active agents,” “No evidence capsules,” “No release candidate,” or “Waiting for event stream.”

10. **Add accessibility themes.**
    Color must never be the only status carrier.

# The redesigned TUI mental model

The current mental model is approximately:

```text
Look at jobs → inspect logs → maybe retry/delete/pause → check release/cache/tests.
```

The world-class mental model should be:

```text
Mission tells me what matters.
Flow shows causal execution.
Evidence proves what happened.
Agents show who/what is acting.
Actions are previewed, safe, and contextual.
Every blocker has an explanation and a next step.
```

That is the difference between a CI dashboard and an agent-first development command center.

## Bottom line

The current API and TUI are already pointed in a very strong direction, especially with nine operational tabs, a flow model, evidence state, action registry, capability API, release gates, and agent pipeline visibility. The biggest weakness is that the TUI is still acting like a smart polling dashboard instead of a **causal, event-driven, proof-aware command center**.

To make it world class, I would redesign around five principles:

1. **One versioned TUI Gateway API.**
2. **Realtime event stream instead of polling.**
3. **Causal graph instead of job list.**
4. **Proof/evidence as a first-class UX object.**
5. **Every screen shows the next safest action.**

That combination would make JeRyu feel dramatically more intuitive, more agent-native, more trustworthy, and much more powerful than GitHub/GitLab’s current CI/CD interfaces.
