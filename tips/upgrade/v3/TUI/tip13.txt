## Bottom line

The current JeRyu TUI/API is a **strong agent-control-plane prototype**, but it is not yet a world-class operator console. The best pieces to preserve are the typed capability intents, shared action registry, broad TUI tab coverage, release gate matrix, live job/log/evidence concepts, and deterministic TUI capture mode. The big missing piece is a true **real-time, entity-linked, action-first UX**: the TUI should not just show pools, jobs, agents, tests, cache, evidence, and release status. It should continuously answer:

> What is happening? What is blocked? Why? What is the safest next action? What proof is missing? What can I approve, retry, cancel, repair, merge, or release right now?

Right now, the API is too coarse and mostly request/response; the TUI is too static and dashboard-like. I would redesign both around a **streaming event model + world snapshot + intent/action system + contextual inspector**.

## What the current API and TUI already have

### Current TUI shape

The CLI exposes `jeryu tui` with one-shot capture support, specific tabs, output sizing, and capture targets including mission, release, jobs, agents, tests, pools, cache, evidence, and secrets. That is a very good foundation because it means the TUI can be regression-tested visually and used in demos/docs without an interactive terminal. ([GitHub][1])

The TUI runtime uses Ratatui-style terminal rendering, raw mode, alternate screen, mouse capture, a background sync loop, and a deterministic PNG capture renderer. The event loop already supports tab navigation, command palette activation, log opening, retry/delete/pause actions, audit toggling, test view toggles, and number-key tab switching. ([GitHub][2])

The current tabs are:

```text
Mission
Release
Jobs
Agents
Tests
Pools
Cache
Evidence
Secrets
```

That is the right top-level information architecture for an agent-first Git system. The current `TuiStateSnapshot` is also broad: it contains pools, GitLab readiness, active containers, recent jobs, pipelines, flow state, logs, cache metrics, release status, test bottlenecks, evidence, secret audit events, agent pipelines, and recent audit events. ([GitHub][3])

### Current visual model

The UI has a header/content/event-console/footer structure. The footer exposes key hints, the event console shows recent evidence/event-like records, and the main area switches by active tab. The TUI already uses semantic colors for job/release states such as success, running, failed, pending, canceled, canary, blocked, and released. ([GitHub][4])

The Mission tab is currently a 2x2 dashboard: system health, pipeline summary, release status, and cache health. That is a reasonable “version 1” cockpit, but it is still mostly passive status rather than an action-oriented operational view. ([GitHub][4])

The Release tab has one of the strongest existing UI ideas: a gate matrix covering main branch, release identity, release pipeline, canary health, E2E gate, and production promotion, with an inspector showing SHA/ref/canary URL/note/eligibility. ([GitHub][4])

The Jobs tab is also comparatively mature. It has a pipeline navigator, flow board/job list/log area, job inspector, evidence capsule details, and contextual actions such as retry/delete plus release-promotion blocker information. ([GitHub][4])

The Agents tab is much less complete. The source itself labels it as a placeholder, then renders agent pipeline branches and a capability/audit history area. That means the UI has the concept of agent visibility, but not yet a true agent cockpit. ([GitHub][4])

### Current API shape

The current capability API is a tagged structured enum called `AgentIntent`. It supports intents such as `ProposePatch`, `RacePatches`, `RunTests`, `FetchCapsule`, `RequestMerge`, `ExplainBlockers`, `GetSystemSnapshot`, `ListAllowedActions`, and `PlanValidation`. That is one of the strongest design choices in the repo: agents should send typed intents, not raw shell strings. ([GitHub][5])

The capability server currently runs over a Unix Domain Socket, reads a bounded request, deserializes an `AgentIntent`, executes it, and writes a JSON response. The response envelope is simple: `success`, `message`, and optional `data`. ([GitHub][5])

The action registry is another strong piece. It models action IDs, labels, key hints, risk tiers, surfaces, dry-run support, descriptions, side-effect classes, required grants, and contract JSON. It already distinguishes local state, Git writes, CI execution, merge, and production-grade side effects. ([GitHub][6])

## What is missing from the current API/TUI

### 1. The API is not real-time enough for a world-class TUI

The current capability API is mostly “send intent, get response.” That works for simple automation but not for a live mission-control UI. The TUI needs a durable event stream:

```text
job.started
job.log.appended
job.failed
pipeline.critical_path.changed
agent.intent.requested
agent.tool.started
agent.tool.finished
merge_gate.blocked
evidence.created
cache.tainted
secret.accessed
release.canary.degraded
```

Right now, the TUI compensates with polling/background sync. The app code polls live logs, periodically refreshes state, checks cache health, and computes some heuristic pipeline timing. The source even notes that some expensive storage checks are proof-of-concept behavior that should be cached less frequently. ([GitHub][3])

A world-class TUI should feel alive without brute-force polling.

### 2. The response envelope is too thin

This current response shape:

```json
{
  "success": true,
  "message": "...",
  "data": {}
}
```

is too limited for rich UI. It should include:

```json
{
  "schema_version": "2026-04-27",
  "request_id": "...",
  "intent_id": "...",
  "status": "accepted|running|succeeded|failed|blocked|requires_approval",
  "message": "...",
  "data": {},
  "errors": [],
  "warnings": [],
  "links": [],
  "entity_refs": [],
  "evidence_ids": [],
  "next_actions": [],
  "event_cursor": "...",
  "expires_at": "..."
}
```

The TUI needs IDs, cursors, links, evidence references, warnings, next actions, and typed failure reasons. Without those, it has to infer too much.

### 3. `RequestMerge` is not good enough for a serious merge UI

The current `RequestMerge` path is especially important because it defines whether the TUI can honestly say “safe to merge.” The implementation currently builds a merge-gate input with weak/fake fields such as `head_sha: None`, hardcoded job counts, trusted tier, and policy flags that do not require successful jobs, no pending jobs, or no recent failures. ([GitHub][5])

For the TUI, this means the merge state cannot be treated as authoritative. A world-class UI needs an exact merge proof:

```text
MR IID
source branch
target branch
exact head SHA
latest pipeline for that SHA
required jobs
failed jobs
pending jobs
test-selection confidence
cache taints
secret-scan status
dependency risk
security scan status
approval state
policy version
evidence bundle ID
```

The UI should never show a green merge state unless the API can prove it from real state.

### 4. `RacePatches` is a great idea but incomplete as a UX primitive

The API supports racing multiple patch hypotheses, but the implementation tells the caller to poll pipeline status to determine a winner and notes that losing branch cleanup or `PollRaceResult` still needs to be implemented. ([GitHub][5])

For TUI UX, patch racing should be one of the flagship agent-first features. It needs:

```text
race created
hypotheses launched
candidate branches
candidate pipelines
scoreboard
winner selected
losers archived/deleted
final evidence bundle
```

Right now, the idea exists, but the TUI cannot yet make it feel magical.

### 5. `PlanValidation` is too coarse

The current `PlanValidation` path is global and mostly checks selector misses in a recent window. It even ignores `project_id` in the implementation. ([GitHub][5])

For the TUI, plan validation should explain:

```text
what changed
which files/symbols/tests are affected
which tests are selected
which tests are skipped
why each was selected/skipped
confidence score
known selector misses
historical false negatives
required extra proof
```

“Valid: true/false” is not enough. The TUI needs explainable validation.

### 6. The Agents tab is not yet an agent cockpit

The Agents tab is currently closer to “agent-related pipelines and audit records” than “what my autonomous workers are doing.” The source marks it as a placeholder. ([GitHub][4])

A real agent cockpit needs:

```text
agent identity
current task
current branch
current MR
current phase
current tool call
current command
last thought/summary
last file touched
capabilities granted
capabilities requested
tokens/cost/time
sandbox ID
network/secret access
current blocker
confidence
kill / pause / resume / reassign actions
```

### 7. The Mission tab is too passive

The current Mission tab has useful panels, but it mostly shows system health, pipeline counts, release status, and cache health. ([GitHub][4])

A world-class Mission tab should instead answer:

```text
Can we ship?
What is blocking us?
Which agent is stuck?
Which proof is missing?
Which action has highest leverage?
What is risky right now?
What changed in the last minute?
```

Counts are secondary. Decisions are primary.

### 8. The TUI lacks a universal contextual inspector

The Jobs tab has an inspector, but the whole TUI should be built around a persistent right-side inspector that changes based on selection:

```text
selected agent → agent cockpit
selected job → logs/evidence/retry
selected MR → merge proof/risk/missing gates
selected test → history/flakiness/affected files
selected cache event → key/provenance/taint/fix
selected release → canary/SLO/rollback
selected capability grant → actor/scope/expiry/revoke
```

The current UI has inspectors in some places, but not a unified interaction model.

### 9. The visual language is too basic

The existing UI uses terminal colors for state, but a top-tier TUI needs a full visual grammar:

```text
state badges
risk badges
progress bars
sparklines
histograms
DAG lines
heatmaps
mini timelines
severity icons
confidence meters
stale-data indicators
```

The current color mappings are useful, but they are mostly status color helpers rather than a complete design system. ([GitHub][4])

### 10. The command palette is not yet the central product surface

The action registry already knows about actions, risk tiers, side effects, required grants, dry-run support, and surfaces. ([GitHub][6])

That should become the heart of the TUI. The command palette should not just be a shortcut list. It should be the operator’s primary interface:

```text
retry failed safe jobs
cancel superseded pipelines
spawn repair agent
explain blocker
run missing proof
request merge dry-run
approve merge
promote canary
rollback release
revoke agent grant
quarantine cache key
```

Every action should have a dry-run preview, blast-radius summary, required grant, expected evidence, and rollback/cancel behavior.

## Ground-up redesign: API first

To make the TUI world-class, I would redesign the API into five layers.

---

### Layer 1: World snapshot API

The TUI should start from one canonical snapshot:

```http
GET /v1/snapshot
```

Response:

```json
{
  "schema_version": "1.0",
  "generated_at": "2026-04-27T18:00:00Z",
  "event_cursor": "evt_...",
  "system": {
    "gitlab": {"status": "ok", "latency_ms": 12},
    "runners": {"online": 6, "busy": 4, "idle": 2},
    "cache": {"status": "ok", "hit_ratio": 0.87},
    "security": {"status": "warning", "open_findings": 2}
  },
  "work_items": [],
  "agents": [],
  "pipelines": [],
  "jobs": [],
  "merge_requests": [],
  "test_plans": [],
  "releases": [],
  "evidence": [],
  "next_actions": []
}
```

The current `GetSystemSnapshot` exists, but it should become a much richer world model that includes all entities the TUI needs to render without guessing. ([GitHub][5])

---

### Layer 2: Event stream API

Add:

```http
GET /v1/events?cursor=...
```

or over Unix socket:

```json
{
  "type": "subscribe",
  "cursor": "evt_123",
  "filters": {
    "project_id": 1,
    "domains": ["agents", "jobs", "release", "security"]
  }
}
```

Event shape:

```json
{
  "cursor": "evt_124",
  "timestamp": "2026-04-27T18:00:01Z",
  "severity": "info|success|warning|critical",
  "domain": "agent|job|pipeline|test|cache|release|security|merge",
  "event_type": "job.failed",
  "actor": {"type": "agent", "id": "agent-7"},
  "entity_refs": [
    {"type": "job", "id": "123"},
    {"type": "pipeline", "id": "456"},
    {"type": "mr", "iid": "14"}
  ],
  "title": "Unit tests failed",
  "detail": "2 tests failed in src/cache.rs",
  "next_actions": [
    {"id": "explain_blockers", "label": "Explain failure"},
    {"id": "spawn_repair_agent", "label": "Spawn repair agent"}
  ]
}
```

This would let the TUI update instantly instead of relying on periodic refreshes and heuristic polling.

---

### Layer 3: Intent API

Keep the current `AgentIntent` idea, but turn it into a durable intent system:

```http
POST /v1/intents
GET  /v1/intents/{intent_id}
GET  /v1/intents/{intent_id}/events
POST /v1/intents/{intent_id}/cancel
```

Intent shape:

```json
{
  "intent_type": "run_tests",
  "idempotency_key": "mr-14-proof-v3",
  "actor": {
    "type": "human|agent|service",
    "id": "operator"
  },
  "subject": {
    "type": "merge_request",
    "id": "14"
  },
  "dry_run": true,
  "payload": {
    "scope": "impacted",
    "ref": "agent/fix-cache"
  }
}
```

Intent response:

```json
{
  "intent_id": "int_123",
  "status": "requires_approval",
  "risk_tier": "medium",
  "side_effect_class": "ci_execution",
  "required_grants": ["agent_task"],
  "blast_radius": [
    "Will create CI pipeline on ref agent/fix-cache",
    "Will consume runner capacity",
    "Will write evidence record"
  ],
  "estimated_cost": {
    "runner_minutes": 8,
    "cache_bytes": 120000000
  },
  "next_actions": [
    {"id": "approve_intent", "label": "Approve and run"},
    {"id": "edit_scope", "label": "Change test scope"}
  ]
}
```

This builds directly on the current action registry concepts: risk tier, side-effect class, grants, dry run, and surfaces. ([GitHub][6])

---

### Layer 4: Entity graph API

Everything in the TUI should be linked:

```text
work item
 → agent task
   → branch
     → MR
       → pipeline
         → jobs
           → logs
           → evidence
       → test plan
       → merge proof
       → release candidate
```

Add entity endpoints:

```http
GET /v1/work-items
GET /v1/agents
GET /v1/agents/{id}
GET /v1/mrs
GET /v1/mrs/{iid}/risk
GET /v1/mrs/{iid}/merge-proof
GET /v1/pipelines/{id}/graph
GET /v1/jobs/{id}
GET /v1/jobs/{id}/trace?cursor=...
GET /v1/test-plans/{id}
GET /v1/evidence/{id}
GET /v1/cache/events
GET /v1/security/posture
GET /v1/releases/{id}
```

The current TUI already tracks jobs, pipelines, evidence, release state, cache, secrets, and agents in its snapshot; the redesign should make those first-class API entities instead of incidental fields in one app state struct. ([GitHub][3])

---

### Layer 5: Explanation and recommendation API

This is the API layer that makes the TUI feel intelligent:

```http
GET /v1/explain/blocker?entity=mr:14
GET /v1/explain/job/123
GET /v1/explain/test-plan/abc
GET /v1/next-actions?scope=mission
GET /v1/risk/mr/14
GET /v1/cache/explain?key=...
GET /v1/release/123/readiness
```

Every explanation should return:

```json
{
  "summary": "Merge is blocked because integration tests are still pending.",
  "blocking_facts": [
    "Pipeline 456 is for the current MR head SHA",
    "Job integration-test is pending",
    "No signed evidence bundle exists for this SHA"
  ],
  "non_blocking_warnings": [
    "Cache hit ratio dropped below 60%"
  ],
  "recommended_actions": [
    {
      "id": "prioritize_job",
      "label": "Prioritize integration-test",
      "risk_tier": "low"
    },
    {
      "id": "run_missing_proof",
      "label": "Run missing proof",
      "risk_tier": "medium"
    }
  ]
}
```

The current `ExplainBlockers` API exists, but it is too narrow. It handles job/release-style explanations and should expand into a universal explanation model for MR, agent, test, cache, release, security, and pipeline blockers. ([GitHub][5])

## Command palette redesign

The command palette should be the TUI’s superpower.

Example:

```text
> explain why MR !14 cannot merge

Result:
MR !14 is blocked by:
1. Integration test job pending
2. Missing signed merge proof
3. Human approval required for medium-risk dependency change

Actions:
[Run missing proof] [Request approval] [Open test plan] [Cancel]
```

Example:

```text
> spawn repair agent for failing job 123

Dry run:
Risk: Medium
Side effects:
- creates branch agent/repair-job-123
- grants run_tests and propose_patch
- writes evidence
Estimated runner cost: 6m
Requires: agent_task grant

[Approve] [Edit scope] [Cancel]
```

This should be backed directly by the action registry, but the registry needs richer schemas, dry-runs, and executable action handlers. The current registry already has action IDs, risk tiers, side effects, grant requirements, and contract JSON, so the foundation is there. ([GitHub][6])

## API changes that would most improve the TUI

### 1. Replace simple capability responses with typed envelopes

Current response is too simple. ([GitHub][5])

New response:

```json
{
  "schema_version": "1.0",
  "request_id": "req_123",
  "intent_id": "int_456",
  "status": "accepted",
  "message": "Test run started",
  "data": {},
  "warnings": [],
  "errors": [],
  "links": [
    {"rel": "pipeline", "href": "/v1/pipelines/789"}
  ],
  "entity_refs": [
    {"type": "pipeline", "id": "789"},
    {"type": "branch", "name": "agent/fix-cache"}
  ],
  "evidence_ids": [],
  "next_actions": [
    {"id": "open_pipeline", "label": "Open pipeline"}
  ],
  "event_cursor": "evt_123"
}
```

### 2. Add a real-time event stream

This is the single most important TUI upgrade.

```http
GET /v1/events?cursor=...
```

or:

```text
jeryu capability subscribe-events
```

The TUI should render from events, not just polling.

### 3. Add action dry-run endpoint

```http
POST /v1/actions/{action_id}/dry-run
POST /v1/actions/{action_id}/execute
```

Dry-run response:

```json
{
  "risk_tier": "medium",
  "required_grants": ["agent_task"],
  "side_effects": [
    "Create branch",
    "Trigger pipeline",
    "Write evidence"
  ],
  "estimated_cost": {
    "runner_minutes": 8
  },
  "rollback": {
    "available": true,
    "description": "Delete branch and cancel pipeline"
  }
}
```

### 4. Add exact merge-proof API

```http
GET  /v1/mrs/{iid}/merge-proof
POST /v1/mrs/{iid}/merge-proof/evaluate
POST /v1/mrs/{iid}/merge
```

A merge proof should include:

```json
{
  "mr_iid": 14,
  "head_sha": "abc1234",
  "target_branch": "main",
  "mergeable": false,
  "blocking_gates": [
    {"gate": "integration_tests", "reason": "pending"}
  ],
  "passed_gates": [],
  "warnings": [],
  "required_approvals": [],
  "evidence_bundle_id": "evb_123",
  "policy_version": "2026-04-27"
}
```

This fixes the current merge-gate weakness. ([GitHub][5])

### 5. Add agent state API

```http
GET /v1/agents

POST /v1/agents/{agent_id}/pause
POST /v1/agents/{agent_id}/resume

```

Agent state:

```json
{

  "status": "testing",

  "current_tool": "cargo test",
  "current_command": "cargo test test_intel::cache",
  "capabilities": [],
  "requested_capabilities": [],
  "sandbox": {},
  "cost": {},
  "blocker": null,
  "next_actions": []
}
```

## What I would keep exactly because it is good

Keep the broad tab set. It is the right mental model. ([GitHub][1])

Keep the typed `AgentIntent` API. It is safer and more agent-friendly than generic shell/API calls. ([GitHub][5])

Keep the action registry as the shared source of truth for TUI, CLI, and capability/API actions. It should become even more central. ([GitHub][6])

Keep the release gate matrix idea. It is one of the most intuitive existing screens. ([GitHub][4])

Keep the job inspector/evidence capsule pattern. It points toward the right “logs plus proof plus action” UX. ([GitHub][4])

Keep deterministic TUI capture. It is rare and very valuable for docs, demos, golden tests, and design iteration. ([GitHub][1])

## My ideal end state

The world-class version of JeRyu’s API/TUI is not “a terminal dashboard for GitLab.” It is an **agent mission-control system**.

The API should expose durable intents, live events, exact proofs, entity graphs, explanations, recommendations, and action dry-runs.

The TUI should feel like:

```text
Datadog + GitHub + GitLab CI + LaunchDarkly + incident command center
```

but specifically designed for autonomous coding agents.

The winning UX principle is:

> Every screen should reduce uncertainty and offer the safest next action.

If JeRyu nails that, the TUI becomes more than a UI. It becomes the human control layer for autonomous software delivery.
