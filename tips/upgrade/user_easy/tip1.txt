I studied the repo through GitHub/raw file access. A local `git clone` attempt failed in this environment because DNS could not resolve `github.com`, so I used the repository pages and structured GitHub file access instead.

## The most important conclusion

The best design is **not** “run the exact same Git command twice against the same working tree.” That would be dangerous and weird: `git commit` would succeed once and fail the second time, `git pull --rebase` could mutate twice, `git reset`/`rebase`/`checkout` could desynchronize state, and scripts could break.

The brilliant version is:

> Install a real `git` PATH shim that always runs the user’s normal Git first.
> If the current repo has `.jeryu/`, the shim then calls JeRyu in “shadow-after” mode.
> JeRyu records the command and mirrors the resulting repo state to the shadow system, rather than blindly replaying every destructive command.

That gives the user exactly the experience they want:

```bash
git status
git commit -m "change"
git push
```

…and behind the scenes:

```text
normal git runs exactly as expected
then JeRyu sees the same command and syncs/observes/shadows it
```

No mental overhead. No special command. No “remember to use jeryu.” No disruption.

## What JeRyu does today

JeRyu is already positioned as a Git-compatible control plane: the README says it wraps Git first, preserves Git muscle memory, provides native wrappers like `jeryu save/sync/undo`, and has `jeryu ship` for origin plus shadow remote syncing. ([GitHub][1])

The current install path is still too manual for the experience you want. The README tells users to clone the repo, run `install.sh`, choose global/local install, and manually add a shell function to `.bashrc`/`.zshrc`. ([GitHub][1]) The repo page also currently reports “No releases published,” so the current path is source-build oriented rather than a polished binary installer. ([GitHub][1])

The current shell shim does this:

```bash
git() {
    if command -v jeryu >/dev/null 2>&1; then
        command jeryu git "$@"
    else
        command git "$@"
    fi
}
```

That replaces normal Git with JeRyu’s Git passthrough; it does **not** currently run both normal Git and a separate JeRyu shadow leg. ([GitHub][1])

The CLI already has a `Git { args }` command, and dispatch currently routes it to `execute_git_passthrough`. ([GitHub][2]) That function simply resolves `JERYU_SYSTEM_GIT` or falls back to `/usr/bin/git`, runs that Git binary with the passed args, and exits with the same status. ([GitHub][3])

Good news: the `.jeryu` config idea is already present. `src/config.rs` has `.jeryu` workspace config parsing and loads files like `.jeryu/shadow.toml`, `.jeryu/proof.toml`, `.jeryu/agent.toml`, and `.jeryu/sandbox.toml`. ([GitHub][4]) Also good: `shadow.rs` already has the right philosophy—its invariants say mirror operations should be idempotent and shadow push failures should not block the primary pipeline. ([GitHub][5]) It also already has repo-local `ensure_remote` and `push_remote` behavior for a named shadow remote. ([GitHub][5])

So the foundation is strong. The missing piece is **activation and interception**.

# The design I would build

## 1. Replace the shell function with a real `git` PATH shim

The shell function is cute, but it will miss exactly the environments you care about: non-interactive agents, scripts, tools launched without `.bashrc`, CI shells, and anything that resolves `git` through `PATH` but not shell functions.

Install this instead:

```text
~/.jeryu/bin/jeryu
~/.jeryu/shims/git
```

Then put this at the front of PATH:

```bash
export PATH="$HOME/.jeryu/shims:$HOME/.jeryu/bin:$PATH"
```

The shim should be a tiny generated executable or script:

```bash
#!/usr/bin/env bash
set -u

REAL_GIT="${JERYU_REAL_GIT:-/usr/bin/git}"

# Hard bypass for recursion or emergencies.
if [ "${JERYU_GIT_BYPASS:-0}" = "1" ]; then
  exec "$REAL_GIT" "$@"
fi

# Discover repo root using real git, never the shim.
REPO_ROOT="$("$REAL_GIT" rev-parse --show-toplevel 2>/dev/null || true)"

# No repo or no .jeryu activation: behave exactly like normal git.
if [ -z "$REPO_ROOT" ] || [ ! -e "$REPO_ROOT/.jeryu" ] || ! command -v jeryu >/dev/null 2>&1; then
  exec "$REAL_GIT" "$@"
fi

# Activated repo: JeRyu owns primary-first orchestration.
export JERYU_REAL_GIT="$REAL_GIT"
export JERYU_GIT_SHIM=1
exec jeryu git --shim "$@"
```

Then `jeryu git --shim …` does the real orchestration:

1. Resolve repo root.
2. Capture before-state: branch, HEAD, dirty/index status, remotes.
3. Run the real Git command with inherited stdin/stdout/stderr.
4. Preserve the real Git exit code.
5. If real Git failed, record the failure and return the same exit code.
6. If real Git succeeded and `.jeryu` is active, run the JeRyu shadow leg.
7. Return the primary Git exit code no matter what JeRyu does, unless the repo explicitly opts into “blocking shadow mode.”

This gives the effortless behavior:

```text
User/agent calls:       git commit -m "fix"
Shim runs:              real git commit -m "fix"
Then JeRyu runs:        jeryu shadow-after commit -m "fix"
User receives:          the real git result
```

## 2. Make `.jeryu/` the activation marker

Activation should be simple:

```text
repo/
  .git/
  .jeryu/
    shadow.toml
```

If `.jeryu/` exists, the repo is JeRyu-aware. If not, Git behaves completely normally.

The current parser already expects `.jeryu/<filename>`, so this fits the codebase instead of inventing a new convention. ([GitHub][4])

I would extend `.jeryu/shadow.toml` from the current minimal fields into something like:

```toml
enabled = true

# The repo’s primary remote remains normal GitHub/GitLab/etc.
primary_remote = "origin"

# JeRyu shadow target. Can be a normal remote name or a host alias.
shadow_remote = "shadow"

# The key idea: mirror final state, do not replay every mutating command.
mode = "mirror-head"

# Human-friendly behavior.
async = true
max_blocking_seconds_on_push = 5
show_status_in_tty = true
quiet_in_scripts = true

# Divergence policy.
on_divergence = "primary_wins"
allow_force_with_lease = true
create_recovery_refs = true

# Optional server alias from ~/.jeryu/hosts.toml
host = "big-runner"
```

Existing fields like `auto_sync_on_commit`, `auto_remediate`, and `max_blocking_seconds_on_push` can remain, but I would add explicit `enabled`, `shadow_remote`, `mode`, `on_divergence`, and `async` fields.

## 3. Shadow by state, not by replay

This is the heart of the design.

For every Git command, JeRyu should receive the command event. But it should not blindly replay every command. It should classify it:

| Command type            | Example                                             | Primary behavior | JeRyu shadow behavior                                 |
| ----------------------- | --------------------------------------------------- | ---------------- | ----------------------------------------------------- |
| Read-only               | `status`, `log`, `diff`, `show`, `rev-parse`        | Run normal Git   | Record/observe only, no shadow mutation               |
| Local index/worktree    | `add`, `restore`, `stash`                           | Run normal Git   | Record event; optionally snapshot dirty state later   |
| Commit/history mutation | `commit`, `merge`, `rebase`, `cherry-pick`, `reset` | Run normal Git   | Mirror resulting `HEAD` to shadow if command succeeds |
| Network sync            | `push`, `pull`, `fetch`                             | Run normal Git   | Push/mirror resulting state to shadow                 |
| Destructive             | `clean`, `reset --hard`, force push                 | Run normal Git   | Mirror final state; create recovery ref if needed     |
| Clone/init              | `clone`, `init`                                     | Run normal Git   | If new repo contains `.jeryu`, auto-attach/doctor     |

So the internal JeRyu mental model becomes:

```text
Primary Git is the source of truth.
JeRyu shadows the resulting truth.
JeRyu never disturbs primary Git unless the repo explicitly opts into enforcement.
```

This avoids the disaster mode where the second command mutates the same working tree twice.

## 4. Make the shadow leg best-effort by default

Default behavior:

```text
primary succeeds, shadow succeeds    -> return primary success
primary succeeds, shadow fails       -> return primary success, mark JeRyu degraded
primary fails                        -> return primary failure, skip shadow mutation
```

For humans, show a short stderr notice only in a TTY:

```text
jeryu: shadow queued; remote runner temporarily unavailable
```

For scripts and agents, keep stdout clean and return the real Git exit code. This is crucial: many tools parse `git status --porcelain`, `git rev-parse`, or `git diff` output. JeRyu must never pollute stdout.

## 5. Add a local event queue

Every activated repo should get a tiny local ledger:

```text
.jeryu/state/
  events.jsonl
  shadow-queue/
  shadow-status.json
```

Or better, use global state under `~/.jeryu/state/repos/<repo-id>/` so `.jeryu` stays clean and commit-friendly.

Each event should record:

```json
{
  "event_id": "uuid",
  "repo_root": "/path/to/repo",
  "command": ["push", "origin", "main"],
  "primary_exit": 0,
  "before_head": "abc...",
  "after_head": "def...",
  "branch": "main",
  "shadow_status": "queued|ok|failed|diverged",
  "created_at": "..."
}
```

This gives you:

```bash
jeryu shadow status
jeryu shadow replay
jeryu shadow doctor
jeryu shadow reconcile
```

And it means shadow failures do not vanish. The user never has to think, but JeRyu can self-heal.

# The install experience

## Local install: one command

Current install is source-build and interactive. I would move to prebuilt release binaries, with source build as fallback. Since the repo currently shows no published releases, that is a product gap to close. ([GitHub][1])

Target install:

```bash
curl -fsSL https://raw.githubusercontent.com/jeppsontaylor/JeRyu/main/install.sh | sh -s -- --yes --git-shadow
```

What it should do:

1. Detect OS/arch.
2. Download prebuilt `jeryu` if available.
3. Fall back to `cargo build --release` only if no binary exists.
4. Resolve the real Git binary before installing the shim.
5. Install `~/.jeryu/bin/jeryu`.
6. Install `~/.jeryu/shims/git`.
7. Add one managed PATH block to `.zshrc`, `.bashrc`, and optionally `.profile`.
8. Run `jeryu doctor --install`.
9. Print one tiny success message:

```text
JeRyu is installed.
Repos with .jeryu/ now get automatic shadow Git.
Normal repos are untouched.
```

## Repo activation: zero surprise

For a repo that already has `.jeryu/`, nothing else is needed.

For a repo that does not:

```bash
jeryu repo init --shadow
```

Or with a remote server:

```bash
jeryu repo init --shadow --host big-runner
```

This writes `.jeryu/shadow.toml`, ensures a `shadow` remote, and runs a doctor check.

## SSH server install: one command

The server install should also be one command:

```bash
jeryu host install ubuntu@big-server --name big-runner --yes
```

Then attach the current repo:

```bash
jeryu repo attach --host big-runner --yes
```

Or combine them:

```bash
jeryu setup --host ubuntu@big-server --repo . --yes
```

What `jeryu host install` should do over SSH:

1. `ssh` into the host.
2. Detect OS, arch, available memory, disk, Docker, systemd, ports.
3. Install Docker if allowed by `--sudo`.
4. Upload or download the matching `jeryu` binary.
5. Create a dedicated service user or use the SSH user.
6. Create a durable data dir.
7. Run `jeryu init --headless`.
8. Install and start `jeryu serve` as a systemd service.
9. Create or expose a GitLab shadow project.
10. Return a host record to the local machine:

```toml
# ~/.jeryu/hosts.toml
[hosts.big-runner]
ssh = "ubuntu@big-server"
gitlab_http = "http://big-server:8929"
gitlab_ssh = "ssh://git@big-server:2224"
runner_class = "large"
```

Then `jeryu repo attach --host big-runner` should:

1. Create/find the shadow project on the remote JeRyu GitLab.
2. Add/update `git remote shadow`.
3. Write `.jeryu/shadow.toml`.
4. Do a first mirror push.
5. Mark status green.

This is the “set and forget” path.

# Important code changes

## `install.sh`

Add:

```bash
--yes
--git-shadow
--install-dir
--no-shell-edit
--real-git
--remote user@host
--uninstall
```

Current `install.sh` prompts for dependencies and install location. That is fine for development, but the product installer should have a fully non-interactive mode. ([GitHub][6])

## `src/commands/git.rs`

Current `execute_git_passthrough` is too simple: it just calls the system Git and exits. ([GitHub][3])

Replace or extend it with:

```rust
pub fn execute_git(args: &[String], mode: GitMode) -> Result<ExitCode>
```

Modes:

```rust
enum GitMode {
    PassthroughOnly,
    ShimPrimaryThenShadow,
    ShadowAfterPrimary,
}
```

Core behavior:

```text
if no .jeryu:
    exec real git
else:
    run real git
    if success:
        run shadow_after()
    return real git exit code
```

## `src/git_bin.rs`

Create a single resolver:

```rust
pub fn real_git() -> PathBuf
```

It should use:

1. `JERYU_REAL_GIT`
2. `~/.jeryu/git-path`
3. platform lookup
4. hard fallback

Then update every internal `Command::new("git")` call to use this resolver or set `JERYU_GIT_BYPASS=1`. This matters because `shadow.rs` currently invokes `"git"` directly in several places; once a PATH shim exists, those calls can accidentally recurse into JeRyu. ([GitHub][5])

## `src/config.rs`

Extend `ShadowConfig`:

```rust
pub struct ShadowConfig {
    pub enabled: bool,
    pub shadow_remote: String,
    pub primary_remote: String,
    pub mode: ShadowMode,
    pub async_: bool,
    pub auto_sync_on_commit: bool,
    pub auto_remediate: bool,
    pub max_blocking_seconds_on_push: u64,
    pub on_divergence: DivergencePolicy,
    pub allow_force_with_lease: bool,
    pub create_recovery_refs: bool,
}
```

Keep backward compatibility: missing fields default safely.

## `src/shadow.rs`

Lean into what it already says: idempotent mirror operations, primary pipeline not blocked by push failures. ([GitHub][5])

Add:

```rust
pub fn shadow_after_git_command(repo, args, before, after) -> Result<ShadowOutcome>
pub fn classify_git_command(args) -> GitCommandKind
pub fn mirror_head(repo, remote, branch, policy) -> Result<ShadowOutcome>
pub fn reconcile(repo) -> Result<ReconcileOutcome>
```

Use `push_remote` for the simplest case, but harden divergence handling around it. ([GitHub][5])

## CLI additions

Add:

```bash
jeryu repo init
jeryu repo attach --host <name>
jeryu repo doctor
jeryu shadow status
jeryu shadow reconcile
jeryu host install <ssh>
jeryu host doctor <name>
jeryu install shim
jeryu uninstall shim
```

There is already a `HostCommands` area, but it currently focuses on storage/doctor/reclaim. I would extend it into remote provisioning.

# How to handle disagreements between primary and shadow

This needs a crisp contract.

## Rule 1: primary always wins

The user’s normal Git remote and local repo are authoritative. JeRyu shadow exists to observe, validate, run agents, and accelerate—not to surprise the user.

## Rule 2: shadow failure never changes the primary Git exit code by default

If `git push` succeeds and the shadow push fails, the user should not see their push become “failed.” Instead:

```text
primary: success
shadow: queued/degraded
exit: 0
```

Then:

```bash
jeryu shadow status
```

shows:

```text
origin/main:  abc123 pushed
shadow/main:  behind by 1 commit
status:       queued retry
reason:       ssh timeout
```

## Rule 3: divergence is repaired by object identity

If origin has `abc123` and shadow has `def999`, JeRyu should not guess by replaying commands. It should compare refs and commit graph:

```bash
git merge-base --is-ancestor shadow/main HEAD
```

Then:

| Situation                                 | Action                                                    |
| ----------------------------------------- | --------------------------------------------------------- |
| Shadow behind primary                     | Fast-forward shadow                                       |
| Shadow has extra commits created by JeRyu | Save recovery ref, then force-with-lease to primary       |
| Shadow has unknown human commits          | Mark diverged; do not overwrite unless config allows      |
| Shadow remote unreachable                 | Queue retry                                               |
| Shadow project missing                    | Auto-create if host credentials allow; otherwise degraded |
| Branch detached                           | Push to `refs/jeryu/detached/<sha>` or observe only       |

## Rule 4: never destroy evidence

Before any force repair, create a recovery ref:

```text
refs/jeryu/recovery/<branch>/<timestamp>
```

Then force-with-lease only if the lease matches what JeRyu saw.

## Rule 5: blocking mode is explicit

Default should be advisory. But for teams that want JeRyu required:

```toml
block_on = ["push"]
max_blocking_seconds_on_push = 10
```

Even then, be careful: once normal `git push` succeeds, you cannot honestly “undo” it just because shadow failed. So blocking mode should run a **shadow preflight** before the primary push, then do a bounded post-push mirror.

# Problems to avoid

## 1. Recursion

Once `~/.jeryu/shims/git` exists, any internal `Command::new("git")` can call the shim again. This is the biggest implementation trap.

Mitigation: install-time real Git capture plus `JERYU_REAL_GIT`, `JERYU_GIT_BYPASS=1`, and a single `real_git()` resolver used everywhere.

## 2. Breaking scripts that parse Git output

Commands like these must remain pristine:

```bash
git status --porcelain
git rev-parse HEAD
git diff --name-only
```

Mitigation: JeRyu never writes to stdout in shim mode. Human messages go to stderr only when stderr is a TTY. In non-TTY mode, log to JeRyu state only.

## 3. Running destructive commands twice

Literal replay is the wrong model.

Mitigation: classify commands and mirror final state. “Every Git command gets a JeRyu shadow event” does not mean “every Git command is destructively replayed.”

## 4. Credential prompts in the shadow leg

A background shadow push that prompts for credentials will hang agents.

Mitigation: set `GIT_TERMINAL_PROMPT=0` for shadow operations. Require `jeryu repo doctor`/`host doctor` to validate credentials up front.

## 5. Malicious `.jeryu` configs

A repo-local config should not be allowed to run arbitrary commands.

Mitigation: `.jeryu` is data only. No shell hooks. No arbitrary scripts. Remote host aliases are resolved from trusted user-level `~/.jeryu/hosts.toml`, not from repo-stored secrets. Never store tokens in the repo.

## 6. Slow Git startup

A shim that runs `git rev-parse` before every Git command adds overhead.

Mitigation: keep detection tiny; use the real Git binary; cache repo activation by cwd/repo root; skip JeRyu entirely when no `.jeryu` exists.

## 7. Agents using absolute `/usr/bin/git`

A PATH shim cannot intercept absolute paths.

Mitigation: for agent environments, provide:

```bash
eval "$(jeryu env --agent)"
```

or configure the agent workspace to prepend the shim PATH. Absolute binary interception is not worth trying to solve globally.

## 8. Server install under the wrong home directory

Current config conventions use `~/.jeryu`. That is fine locally, but services need deterministic state paths.

Mitigation: add `JERYU_HOME` or `JERYU_DATA_DIR`, then use `/var/lib/jeryu` for server installs.

# What will work especially well

The repo already has most of the right primitives:

JeRyu already has a `jeryu git` surface and native Git wrappers, so the user-facing “Git-compatible layer” is already part of the product identity. ([GitHub][1]) The existing dispatch route means there is a clean place to replace simple passthrough with primary-plus-shadow orchestration. ([GitHub][2])

`.jeryu` workspace config is already a pattern in the codebase, so using `.jeryu/` as the activation marker is a natural extension, not a new mental model. ([GitHub][4])

The shadow module already thinks in terms of idempotent mirroring and non-blocking shadow failures, which is exactly the right philosophy for a frictionless shadow Git layer. ([GitHub][5])

The missing magic is just: install a robust shim, make it repo-activated, make `jeryu git` primary-first, and make shadow sync state-based instead of command-replay-based.
