I wasn’t able to `git clone` inside the sandbox because DNS resolution failed there, but I did pull the relevant repo files through GitHub access and studied the install, CLI, Git passthrough, `.jeryu` config, bootstrap, and shadow paths.

## The key design decision

Do **not** literally run every mutating Git command twice against the same worktree.

That sounds attractive, but it breaks fast: `git checkout -b x` fails the second time, `git commit` either creates a duplicate attempt or fails, `git pull --rebase` can produce weird second-order effects, and `git reset --hard` twice is a terrifying footgun.

The brilliant version is:

> Every `git …` invocation is intercepted. Normal Git runs exactly once. If the repo has `.jeryu`, JeRyu observes the successful result and mirrors/syncs the resulting state into the JeRyu shadow world.

That gives the human and agent the experience you want: they type only `git …`; normal Git remains normal; JeRyu works invisibly.

## What JeRyu already has

JeRyu is already positioned as a single-binary Rust control plane that wraps Git first and later replaces more of the workflow. The README explicitly describes a passthrough layer where `git status` / `git commit` flows through JeRyu, native wrappers like `jeryu save`, `jeryu sync`, `jeryu undo`, and `jeryu ship`, and `ship` is intended to push both origin and a local `shadow` CI remote. ([GitHub][1])

Today’s install is not yet the “set and forget” version. The README asks users to clone the repo, run `install.sh`, then manually paste a shell function into `~/.bashrc` or `~/.zshrc`; the current script builds from source with Cargo, asks where to install, and prints the manual shell shim afterward. ([GitHub][1]) 

The CLI already has the right raw ingredients: `jeryu git …` as a passthrough command, `save/sync/undo/ship`, shadow sync management, and shadow remote management.  Dispatch currently maps `jeryu git` directly to `execute_git_passthrough`, and `jeryu serve` starts the shadow loop alongside the main engine. 

The current `execute_git_passthrough` runs a real Git binary using `JERYU_SYSTEM_GIT` or `/usr/bin/git`, then exits with Git’s status. `ship` already does the “origin first, shadow second, shadow failure does not block” behavior. 

JeRyu already has repo-local `.jeryu` parsing for files like `.jeryu/shadow.toml`, `.jeryu/proof.toml`, `.jeryu/agent.toml`, and `.jeryu/sandbox.toml`. The existing `ShadowConfig` includes `auto_sync_on_commit`, `auto_remediate`, and `max_blocking_seconds_on_push`, defaulting to 5 seconds. 

The shadow worker is also close: it stores source repo configs, polls every 2 seconds, skips unchanged HEADs, uses exponential backoff after failures, pushes to the GitLab shadow project, and can optionally push upstream via global settings. 

For remote runners, `jeryu bootstrap` already tries to do headless setup: generate secrets, write Docker Compose, start GitLab/Vault/Postgres, create a GitLab PAT, create runner pools, and run a smoke test.

# Best design: the repo-scoped invisible Git shim

Install a real executable named `git` earlier in `PATH`. That shim should activate only when the current command is inside a repo containing `.jeryu`.

The flow should be:

```text
agent / human
    ↓
git status
    ↓
~/.jeryu/bin/git   # symlink or tiny shim
    ↓
jeryu git --shim status
    ↓
detect real git + repo root + .jeryu
    ↓
run real git status exactly once
    ↓
if .jeryu active, record/trigger safe JeRyu shadow action
```

For non-JeRyu repos, the shim should `exec` the real Git immediately. No logging, no wrapping, no surprises.

For JeRyu repos, the shim should run the real Git command with inherited stdin/stdout/stderr so interactive Git still feels native. Only after the real command exits successfully should JeRyu do shadow work.

## The installed shape

The install should create:

```text
~/.jeryu/bin/jeryu
~/.jeryu/bin/git -> ~/.jeryu/bin/jeryu
~/.jeryu/settings.json
~/.jeryu/servers/
~/.jeryu/logs/
```

Then prepend `~/.jeryu/bin` to the user’s shell PATH with an idempotent block.

Even better: make the `jeryu` binary inspect `argv[0]`. If it was invoked as `git`, it enters shim mode before initializing the heavy async runtime. That gives you one binary, one symlink, and no duplicate shim binary.

```rust
// conceptual only
fn main() -> Result<()> {
    if invoked_as("git") {
        return jeryu::shim::run_git_shim();
    }

    run_normal_jeryu_cli()
}
```

This is much better than a shell function because it works for agents, scripts, non-interactive shells, subprocesses, and ordinary CLI usage. The shell function in the README is a good prototype, but it is too fragile for “effortless shadow mode.” ([GitHub][1])

# The new install UX

## Local install

Ideal command:

```bash
curl -fsSL https://jeryu.dev/install | sh
```

What it should do without asking the user to think:

```text
1. Detect OS + arch.
2. Download a prebuilt signed jeryu release.
3. Verify checksum/signature.
4. Resolve the real Git path before installing the shim.
5. Install jeryu to ~/.jeryu/bin/jeryu.
6. Install ~/.jeryu/bin/git as a symlink/shim.
7. Add ~/.jeryu/bin to PATH idempotently.
8. Save real_git in ~/.jeryu/settings.json.
9. Run jeryu doctor --shim.
```

Current install builds from source and asks the user questions; that should become a fallback/dev path, not the default. 

## Repo activation

Once the shim is installed, the repo controls activation.

A minimal `.jeryu/shadow.toml`:

```toml
[shim]
enabled = true
quiet = true
failure_policy = "warn" # warn | strict | silent

[shadow]
enabled = true
auto_sync_on_commit = true
sync_on_push = true
remote = "shadow"
branch = "same"
max_blocking_seconds_on_push = 5
force_shadow = true
```

Then the user or agent just runs:

```bash
git status
git commit -m "ship the thing"
git push
```

No `jeryu` command has to be in the agent’s prompt or tool list.

# What each Git command should do

| Git command type                              | Primary behavior          | JeRyu shadow behavior                                                      |
| --------------------------------------------- | ------------------------- | -------------------------------------------------------------------------- |
| `status`, `diff`, `log`, `show`               | Run real Git only         | Optionally update health cache; do not duplicate output                    |
| `add`, `restore --staged`, index-only changes | Run real Git              | Record event; no shadow push needed                                        |
| `commit`                                      | Run real Git once         | If success and `auto_sync_on_commit=true`, push resulting HEAD to shadow   |
| `merge`, `rebase`, `pull`                     | Run real Git once         | If success and HEAD changed, shadow sync; if conflicts/failure, do nothing |
| `push`                                        | Run real Git/origin first | If primary push succeeds, push same resulting ref to shadow                |
| `checkout`, `switch`, `reset`                 | Run real Git once         | Reconcile shadow only if configured and safe                               |
| `clone`                                       | Run real Git              | After clone, if target repo has `.jeryu`, lazily activate                  |
| unknown/destructive commands                  | Run real Git              | Record only; no shadow action unless explicitly classified                 |

The default should be **primary-first, fail-open**:

```text
normal Git failure  => JeRyu does nothing
normal Git success  => JeRyu tries shadow
shadow failure      => normal Git exit code is preserved
```

For teams that want JeRyu to gate pushes, `.jeryu/shadow.toml` can opt into strict mode:

```toml
[shim]
failure_policy = "strict_on_push"
```

# The “call `jeryu git` automatically” implementation

The user-facing behavior can match your phrasing exactly:

```bash
git commit -m "x"
```

Internally:

```text
~/.jeryu/bin/git
  -> jeryu git --shim commit -m x
      -> real git commit -m x
      -> jeryu shadow observe/sync
```

I would refactor the current `execute_git_passthrough` because it exits immediately after real Git. That makes post-processing awkward. Instead, it should return or manage an exit code after the shadow step. The current function’s `std::process::exit(...)` behavior is fine for a pure passthrough, but not for the new shadow orchestration. 

Also: stop defaulting to `/usr/bin/git`. Resolve the real Git path at install time and store it. On macOS/Homebrew, Linux alternatives, Nix, asdf, devcontainers, and CI agents, `/usr/bin/git` is not always the Git the user expects.

# Remote server install via SSH

The killer UX should be one local command:

```bash
jeryu server install ubuntu@big-runner-box --repo . --yes
```

What that command should do:

```text
1. SSH preflight:
   - OS/arch
   - Docker/Compose availability
   - disk/memory
   - ports
   - SSH host key fingerprint
   - systemd user service availability

2. Install JeRyu remotely:
   - download/copy signed jeryu binary
   - install to ~/.jeryu/bin/jeryu
   - create ~/.jeryu/settings.json
   - do not require Rust

3. Bootstrap remote control plane:
   - jeryu bootstrap
   - jeryu serve as a systemd user service
   - create/ensure GitLab project for this repo
   - create runner pools

4. Pair local repo:
   - write/update .jeryu/shadow.toml
   - add/ensure git remote shadow
   - store remote server metadata outside the repo
   - run a smoke push

5. Print one final line:
   - “Done. Your agent can use normal git.”
```

The remote server should own secrets. The repo should contain only non-secret routing config. Local secrets and server tokens belong in `~/.jeryu/servers/<server-id>.json` or remote `~/.jeryu/jeryu.env`, not in `.jeryu`.

A generated repo config could look like:

```toml
[shim]
enabled = true
quiet = true
failure_policy = "warn"

[shadow]
enabled = true
auto_sync_on_commit = true
sync_on_push = true
remote = "shadow"
branch = "same"
max_blocking_seconds_on_push = 5
force_shadow = true

[server]
name = "big-runner-box"
host = "ubuntu@big-runner-box"
project_id = 2
git_ssh_url = "ssh://git@big-runner-box:2224/root/my-repo.git"
```

For public remote servers, I would change the remote bootstrap defaults to bind sensitive services to localhost or firewall them. Current generated Compose exposes Vault on `0.0.0.0`, which is fine for local dev but not the right default for an internet-accessible runner host.

# Handling repo disagreement

The invariant should be:

> Origin / normal Git is the source of truth. Shadow is disposable, reproducible, and never allowed to damage the primary repo.

## If normal Git fails

Do not run shadow.

Example:

```text
git push origin main  # rejected
JeRyu shadow push     # skipped
```

Reason: shadow should not make failed primary state look accepted unless explicitly configured.

## If normal Git succeeds but shadow fails

Preserve the real Git exit code. Queue the shadow event for retry. Show at most a compact warning:

```text
jeryu: shadow sync queued; remote unavailable
```

No duplicate command output. No scary wall of logs.

## If shadow branch diverges

Use this policy:

```text
1. Fetch shadow branch.
2. Compare shadow HEAD with last_pushed_sha from JeRyu state.
3. If shadow only diverged from JeRyu-owned commits:
      force-with-lease update shadow.
4. If shadow contains unknown/human commits:
      push current state to refs/heads/jeryu/reconcile/<branch>/<timestamp>
      mark repo degraded
      never overwrite unknown commits silently.
```

For isolated CI shadow branches, force updating is usually correct. For any upstream/human remote, never force unless the config explicitly allows it.

## If origin and shadow have different test results

Do not change normal Git behavior by default.

Record:

```text
primary_sha
shadow_sha
origin_push_status
shadow_push_status
pipeline_id
failure capsule
```

Then expose:

```bash
jeryu next
jeryu shadow status
jeryu explain-blocker ...
```

The current code already has `next`, `progress`, `explain-blocker`, shadow status, and failure evidence concepts that can support this nicely.

# Problems to anticipate and how to avoid them

| Problem                              | Why it matters                                                     | Mitigation                                                                                        |
| ------------------------------------ | ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------- |
| Infinite recursion                   | `git` shim calls `jeryu`, which calls `git`, which hits shim again | Resolve real Git at install; set `JERYU_REAL_GIT`; set `JERYU_SHIM_DEPTH=1`; direct exec real Git |
| Mutating command run twice           | Many Git commands are not idempotent                               | Run primary Git once; JeRyu mirrors resulting state, not the same command                         |
| Shell function does not catch agents | Agents often run non-interactive subprocesses                      | Install an actual `git` executable/symlink earlier in PATH                                        |
| Absolute `/usr/bin/git` bypass       | Some tools call absolute Git                                       | Cannot fully intercept; add `jeryu doctor`, optional hooks fallback for commit/push               |
| Slow `git status`                    | Developers will remove the shim if it feels slow                   | No `.jeryu` = direct exec; `.jeryu` read-only commands = no shadow network calls                  |
| Shadow failure breaks work           | Users lose trust immediately                                       | Default fail-open; strict mode only by config                                                     |
| Duplicate/noisy output               | Agents may parse Git output                                        | Shadow side logs quietly; warnings only on failure                                                |
| Concurrent agents                    | Multiple Git commands can race                                     | Use `.git/jeryu/queue` plus lock file; coalesce by final HEAD                                     |
| Rebase/merge conflicts               | Shadowing half-finished state is dangerous                         | Only shadow after primary success and clean HEAD                                                  |
| Detached HEAD                        | Current shadow remote push can require explicit branch             | Use configured branch or `refs/heads/jeryu/detached/<sha>`                                        |
| Worktrees/submodules                 | Wrong root detection causes wrong config                           | Use real Git’s `rev-parse --show-toplevel` and `--git-common-dir`                                 |
| Token leakage                        | Current HTTP PAT-in-URL style can appear in process lists/logs     | Prefer SSH deploy keys or credential helpers; redact logs; avoid PAT in command args              |
| Malicious `.jeryu`                   | Cloned repo could try to exfiltrate code                           | `.jeryu` must be schema-only, no arbitrary commands; require trust for new external hosts         |
| Remote server exposed ports          | Internet runner host can expose GitLab/Vault                       | SSH host-key pinning, firewall/preflight, localhost binds for sensitive services                  |
| PATH not active in existing agent    | The agent may already be running                                   | `jeryu doctor --agent`; optional repo hooks fallback; clear “restart agent shell” diagnostic      |

# What I would build first

## Phase 1: Invisible local shim

Add:

```text
src/shim.rs
src/git_real.rs
src/git_classify.rs
```

Commands:

```bash
jeryu shim install
jeryu shim doctor
jeryu shim uninstall
```

Behavior:

```text
no .jeryu      -> exec real git
.jeryu exists  -> run real git once, then JeRyu shadow observe
```

## Phase 2: Repo auto-activation

Add:

```bash
jeryu repo activate --auto
jeryu repo doctor
jeryu shadow reconcile
```

The first time the shim sees `.jeryu`, it should lazily ensure:

```text
.git/jeryu/
.git/jeryu/events.jsonl
.git/jeryu/state.json
shadow remote exists if configured
DB shadow_sync_config exists if using local daemon mode
```

## Phase 3: SSH server install

Add:

```bash
jeryu server preflight ubuntu@host
jeryu server install ubuntu@host --repo . --yes
jeryu server doctor ubuntu@host
jeryu server logs ubuntu@host
```

This should use existing `jeryu bootstrap`, then install `jeryu serve` as a service. Bootstrap is already close to headless control-plane setup. 

## Phase 4: Hardening

Add tests for:

```text
- no .jeryu = exact real Git passthrough
- .jeryu = primary Git output and exit code preserved
- primary Git failure = no shadow action
- shadow failure = normal Git exit preserved
- no recursion
- commit/push classification
- rebase conflict skip
- worktree/submodule root detection
- token redaction
- remote install dry-run
```

# The north-star UX

For a repo that already has `.jeryu`:

```bash
curl -fsSL https://jeryu.dev/install | sh
```

Then:

```bash
git status
git commit -m "whatever"
git push
```

And JeRyu quietly shadows it.

For a new repo that needs a big remote runner box:

```bash
curl -fsSL https://jeryu.dev/install | sh
jeryu server install ubuntu@big-runner-box --repo . --yes
```

Then the agent still only needs:

```bash
git commit -m "fix"
git push
```

That is the set-and-forget design: normal Git remains the visible interface, `.jeryu` is the opt-in switch, JeRyu becomes the invisible shadow layer, and remote runners are one SSH bootstrap away.
