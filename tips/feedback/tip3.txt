I reviewed the repo through GitHub/GitHub API file inspection. I could not compile it locally in this sandbox because outbound DNS to GitHub failed, so I am **not** claiming build/test results. This is a careful product + architecture + code review based on the repository files I inspected.

## Bottom-line verdict

JeRyu has a genuinely exciting foundation: Rust single-binary control plane, GitLab runner orchestration, DB-backed state, TUI, MCP/capability surface, evidence/capsule concepts, smart test selection, release gates, Vault/secrets, and agent workflow scaffolding. The repo structure is already broad and serious, with core modules for admission, agents, capabilities, cache, engine, execution, GitLab client, impact, MCP, release, sandbox, shadow sync, state, tests, TUI, and more. ([GitHub][1])

The biggest gap is that the **Git compatibility story is currently much thinner than the mission**. The README promises Git-compatible wrapping, native helpers, and `jeryu ship` pushing both origin and a shadow remote, but the actual `jeryu git` implementation is only a direct shell-out to one Git binary.  

Your new direction is exactly right: **stop thinking of JeRyu as “shadow Git internally,” and instead make JeRyu a parallel, Git-compatible control plane that observes, mirrors, validates, explains, and gates every Git action while preserving normal Git behavior.**

# Proposed mission statement

**JeRyu’s mission is to make every Git action agent-ready. Every command that humans and agents already know—`git status`, `git commit`, `git push`, `git rebase`, `git merge`—must continue to work exactly as Git users expect, while JeRyu runs beside it as an autonomous Rust control plane that records intent, mirrors critical state, launches validation, explains risk, and gives agents bounded authority to repair, verify, and ship software safely. Git remains the trusted ledger; JeRyu becomes the intelligence layer that makes version control observable, programmable, parallel, and AI-native.**

**North star:** **100% Git-compatible at the edge, autonomous and policy-governed underneath. JeRyu should become the default version-control interface for both humans and coding agents.**

This is stronger than the current README framing of “JeRyu wraps Git first, then replaces it.” That phrase is catchy, but it creates the wrong fear: developers do not want their Git replaced before trust is earned. The current MISSION.md already says “Git remains the ledger of truth,” which is the better strategic anchor. 

I would change the public promise to:

> **JeRyu does not ask you to leave Git. It makes Git agent-native.**

## Replace this public story

> “JeRyu wraps Git first, then replaces it.”

## With this public story

> **JeRyu runs beside Git first, proves trust second, and only replaces internals when compatibility, safety, and evidence are stronger than Git alone.**

# Highest-impact technical change: build a real Dual Git Engine

Right now, `src/commands/git.rs` does this:

`jeryu git <args>` resolves `JERYU_SYSTEM_GIT` or defaults to `/usr/bin/git`, runs that binary with the args, and exits with its code. The native helpers call Git subcommands directly, and `ship` sequentially pushes `origin` then `shadow`. 

That is not yet “run all git commands in both the remote and `jeryu git`.” It is a passthrough shim.

The correct architecture is:

```text
git <args>
  ↓ shell shim
jeryu git <args>
  ↓
Dual Git Engine
  ├─ System Git path: run the real git command exactly
  └─ JeRyu path: classify, observe, record, mirror, validate, trigger runners
```

## Critical rule

For mutating commands, **do not run the exact same mutation twice in the same working tree**.

For example:

`git commit` cannot literally run twice.

`git clone` cannot clone twice into the same destination.

`git pull --rebase` cannot safely replay twice.

`git push` can be mirrored to a second target, but not blindly duplicated to the same target.

So the dual engine should not mean “execute identical process twice always.” It should mean:

**Run the canonical Git command once, exactly as Git would; run JeRyu’s corresponding control-plane action in parallel or immediately after, depending on safety.**

# Proposed Dual Git modes

Add a first-class mode model:

```rust
pub enum GitRunMode {
    Observe, // never changes Git behavior; records and explains only
    Mirror,  // mirrors safe remote writes / triggers validation
    Gate,    // may block risky actions by policy
    Agent,   // stricter machine-consumable mode for autonomous workers
}
```

Default should be `Observe` or `Mirror`, not `Gate`.

## Behavior by command class

| Command class         | Examples                                                    |          System Git behavior | JeRyu behavior                                                |
| --------------------- | ----------------------------------------------------------- | ---------------------------: | ------------------------------------------------------------- |
| Read-only             | `status`, `log`, `diff`, `show`, `branch --list`            | Run exactly, preserve output | Record context, update TUI, maybe compute impact              |
| Local mutation        | `add`, `commit`, `reset`, `checkout`, `rebase`, `merge`     |                     Run once | Capture before/after HEAD, worktree state, risk, evidence     |
| Remote read           | `fetch`, `pull`, `clone`, `ls-remote`                       |                     Run once | Record remote refs, detect divergence, update mirror metadata |
| Remote write          | `push`, `tag push`, `push --delete`                         |      Run once to user target | Mirror to verification target if configured; trigger runners  |
| Destructive/high-risk | `reset --hard`, `clean -fdx`, `push --force`, branch delete |        Run under mode policy | Warn/record/gate depending on config                          |
| Unknown               | aliases, custom Git commands                                |                  Run exactly | Record “unknown passthrough”; never block by default          |

This is how you get all Git commands into JeRyu without breaking Git.

# Files to change immediately

## `src/commands/git.rs`

This should become a thin adapter only.

Current problems:

The passthrough function calls one Git binary and exits from inside the command function. The helper commands ignore several failure states, and `ship` prints success even if the primary push fails or the shadow push fails. 

Change it from:

```rust
pub fn execute_git_passthrough(args: &[String]) -> Result<()>
```

To something conceptually like:

```rust
pub async fn execute_git_passthrough(args: Vec<String>) -> Result<ExitCode>
```

Then return the exit code from `main` rather than calling `std::process::exit()` inside library code. This will make it testable and composable.

Specific fixes:

1. Preserve exact Git exit code.
2. Preserve exact stdout/stderr behavior.
3. Do not default to `/usr/bin/git`.
4. Never claim success if Git failed.
5. Do not silently ignore `git add`, `git push`, or `git reset` failures.
6. Emit structured JeRyu event records even when Git fails.
7. Add JSON output mode for agents, but do not alter default Git output.

## `src/dispatch.rs`

The file says dispatch should wire commands to modules without business logic, and Cargo metadata also says `main.rs` should be dispatcher-only and state changes should go through `state::Db`.  

But `dispatch.rs` currently contains a lot of business logic for shadow commands, action listing, next-action calculation, and blocker explanation. 

Move logic into:

```text
src/commands/shadow.rs
src/commands/action.rs
src/commands/next.rs
src/commands/explain.rs
src/commands/mcp.rs
src/commands/capability.rs
```

Then `dispatch.rs` becomes a real dispatcher.

## `src/shadow.rs`

This module is useful, but the public language should change.

Current behavior:

It manages shadow remotes, runs a polling sync loop, pushes to GitLab, optionally pushes an upstream URL, and treats shadow push failures as non-blocking. 

Issues to fix:

1. Uses plain `git` internally instead of the configured system Git path. 
2. Injects PAT into the push URL string, which can leak through process arguments or diagnostics. 
3. Polls every 2 seconds, which is simple but not ideal once the control plane grows.
4. “Shadow” should become “mirror” or “verification target” in docs and user-facing UI.
5. `ShadowPushOutcome = Option<(String, Option<Result<(), String>>)>` is too opaque.

Rename or wrap:

```text
src/shadow.rs         -> keep for compatibility
src/git/mirror.rs     -> new public mirror implementation
```

Deprecate CLI names slowly:

```text
jeryu shadow-remote ...     old, hidden/deprecated
jeryu mirror ...            new
jeryu git mirror status     optional future
```

## `src/settings.rs`

Settings has a major correctness bug.

The file says settings merges should “never silently discard explicit user configuration,” but `load()` logs a warning and returns defaults on parse error. `get()` can also fall back to defaults. 

That violates the stated invariant.

Change behavior:

1. If `settings.json` is corrupt, write `settings.json.bad.<timestamp>`.
2. Fail with a clear error.
3. Offer `jeryu settings repair` or `jeryu settings reset`.
4. In non-interactive agent mode, fail closed.

Also fix leftover branding:

`VaultSettings.prefix` defaults to `"veox"`, and config constants also still use `veox`.  

Change to:

```rust
prefix: "jeryu"
```

## `src/config.rs`

Security-sensitive issue:

The Docker Compose template binds Vault to `0.0.0.0`, while webhook and MCP defaults are loopback.  

Change Vault default to:

```yaml
ports:
  - "127.0.0.1:{vault_port}:8200"
```

Only allow `0.0.0.0` through explicit settings.

Also, workspace config parse failures currently warn and fall back to defaults. For proof, sandbox, and agent policy config, that is too permissive. 

Add:

```rust
pub enum ConfigLoadMode {
    Permissive,
    FailClosed,
}
```

Use `FailClosed` for sandbox, policy, proof, capability, secrets, and agent authorization files.

## `src/main.rs`

Remove `#![allow(dead_code)]` eventually. It hides decay. 

Also update the module comment. “Headless GitLab with Rust god-mode control” is memorable, but it is not the right professional public positioning for a system that should win enterprise and agent trust. 

Use:

> `jeryu — Git-compatible, agent-native version control control plane.`

# Rust TUI improvements

The TUI is one of the strongest differentiators. It should become the **JeRyu cockpit**: not just runner monitoring, but the live view of Git + agents + evidence + risk.

Current TUI strengths:

The app has a broad state snapshot model with pools, GitLab readiness, jobs, pipelines, live logs, shadow summary, cache metrics, release state, test bottlenecks, evidence, secrets, agent pipelines, audit events, runner feeds, and pipeline progress. 

The action registry is a great idea: one source for CLI/TUI/capability actions, risk tiers, surfaces, dry-run flags, and tests for uniqueness/grants. 

Current TUI issues:

# Capability and MCP improvements

The capability layer is promising and already has typed `AgentIntent`, enveloped requests, protocol versioning, nonces, expiry, actor, optional grant, budget, and idempotency fields. 

But several fields are currently more aspirational than enforced.

## Fix capability enforcement

Current request shape includes:

```text
idempotency_key
budget
grant
```

But validation mainly checks protocol version, request ID, actor, nonce, expiry, and replay. 

Add:

1. `validate_grant_for_intent()`
2. `validate_budget_for_intent()`
3. `enforce_idempotency_key()`
4. `reject_legacy_side_effecting_intents_by_default()`

Legacy support is fine for read-only operations, but side-effecting intents such as `ProposePatch`, `RacePatches`, `RunTests`, and `RequestMerge` should require an envelope and grant.

## Make RacePatches complete

Current `RacePatches` launches branches and notes that losing branches need manual cleanup or a future poll result. 

Add:

```text
AgentIntent::PollRaceResult
AgentIntent::CancelRace
AgentIntent::CleanupRace
```

Store race state in DB.

## Improve MCP argument errors

MCP tools currently build intents by returning `Option<AgentIntent>`, so invalid input can become a generic failure instead of a helpful schema/path-specific error. 

Change:

```rust
pub(crate) fn build_intent(&self, args: Value) -> anyhow::Result<AgentIntent>
```

Return errors like:

```text
missing required field: project_id
invalid field test_scope: expected one of unit,integration,lint,full
modifications[2].file_path must be a string
```

Agents need crisp errors.

### 1. `jeryu git` is not yet dual-run

**Fix:** fail with repair path.

Compose binds Vault to `0.0.0.0`. 

**Fix:** default to `127.0.0.1`.

# The product wedge that can make this popular

The winning wedge is not “AI Git replacement.” That sounds scary.

The winning wedge is:

**“Install one shim and every Git command becomes observable, agent-safe, and CI-aware without changing developer muscle memory.”**

That is a killer story.

The first demo should be:

```bash
git status
git commit -m "fix cache invalidation"
git push
jeryu tui
```

And the TUI shows:

```text
Git command: push
System Git: success
JeRyu mirror: success
Validation: running
Risk: low
Evidence: collecting
Agent actions: explain, retry, propose fix
```

That makes the product obvious in 30 seconds.

# Final recommendation

Do **not** delete the runner, GitLab, capability, TUI, evidence, cache, release, and agent work. That is the moat.

But do simplify the mental model:

**JeRyu is not a shadow Git fork. JeRyu is the parallel intelligence/control plane for Git.**

That single pivot clarifies everything:

* `git` remains trusted.
* `jeryu git` becomes the universal capture point.
* runners become automatic validation.
* agents get a safe API.
* TUI becomes the command center.
* “shadow” becomes an implementation detail, not the mission.
* the project can become popular because it does not ask developers to abandon what they already trust.
