I studied the live repo through its GitHub file surfaces and the GitHub connector. I was not able to `git clone` inside the sandbox because outbound DNS resolution failed, so the review below is based on the live repository files, docs, and code paths I could inspect directly.

The biggest recommendation is this: **do not literally run `git <command>` and `jeryu git <command>` as two separate mutating commands.** That would either recurse through the shell shim or run destructive/history-changing commands twice. The winning architecture is: **`jeryu git` becomes the universal dual-lane executor: real Git executes exactly once, while JeRyu records, mirrors, explains, and triggers agents/runners as a sidecar.** The current README shell shim already routes `git "$@"` into `jeryu git "$@"`, and the current `jeryu git` implementation only shells out to system Git and exits with Git’s status.  

## The north-star mission I would put in `MISSION.md`

**JeRyu exists to make Git agent-native without making humans relearn Git.**

**Git remains the universal commit ledger. JeRyu becomes the Rust control plane around it: every Git command stays compatible, while JeRyu records intent, mirrors outcomes, triggers the right proof, explains risk, and gives autonomous agents bounded authority to act.**

**The north star: a developer or agent can type any Git command and trust that JeRyu will run real Git exactly as expected, while automatically deciding what changed, what proof is missing, what work is obsolete, what can be safely skipped, and when escalation is required.**

**JeRyu wins by being invisible when Git should be invisible, and decisive when Git is not enough.**

The existing mission is thoughtful, especially around Git as the ledger of truth, Rust owning orchestration/policy, and agents operating inside explicit capability bounds. But it currently reads more like a CI/control-plane thesis than a crisp “AI-native Git replacement” product promise. It also still frames GitLab as the first substrate and emphasizes missing CI intelligence primitives, which are valuable, but the new north star should lead with universal Git compatibility plus agent-safe automation.

## Executive diagnosis

JeRyu already has surprisingly strong foundations for an ambitious AI-ready Git layer: a Rust single-binary CLI, GitLab bootstrap/control, runner pools, webhook ingestion, state persistence, capability APIs, MCP, VTI smart-test planning, evidence capsules, risk gates, and a Ratatui TUI. The README positions it as a Git-compatible layer that “wraps Git first, then replaces it,” and the architecture docs split the system into interface, control, state, execution, and intelligence planes.  

The current weak point is that the Git compatibility story is still thin. `src/commands/git.rs` is basically a passthrough wrapper around a Git binary. It does not yet record command intent, classify risk, emit events, trigger runners, mirror outcomes, or expose a parity/evidence layer. The “AI-ready Git replacement” promise is therefore ahead of the Git shim implementation. 

There is also a strategic mismatch: public docs, CLI command names, serve startup, and the TUI still emphasize `shadow` remotes and shadow sync, while your new direction is broader and cleaner: automatically observe and process **all** Git commands through `jeryu git`. The README says `jeryu ship` pushes to origin and local `shadow`; `cli.rs` still has `Shadow` and `ShadowRemote` command families; `dispatch.rs` starts `shadow::run_shadow_loop`; and `src/shadow.rs` implements a polling remote-mirroring loop.    

The fastest path to leveling this up is to replace the “shadow remote” product story with a **Universal Git Sidecar**: every Git command is run once through real Git, then JeRyu captures before/after state, records a durable event, classifies risk, triggers the correct runner/agent workflow, and shows parity in the TUI.

## The core architecture change: Universal Git Dual-Run

### What the system should do

`jeryu git <args>` should have two lanes:

**Lane 1: Real Git lane.**
Run the exact system Git command once, preserving Git behavior: current working directory, stdin/stdout/stderr, TTY behavior, pager behavior, environment variables, hooks, exit status, and signal behavior. This lane is sacred. If JeRyu breaks Git muscle memory, adoption dies.

**Lane 2: JeRyu sidecar lane.**
In parallel where safe, and after the command where necessary, JeRyu records command intent/outcome, classifies risk, updates state, mirrors when configured, triggers runners, produces evidence, and exposes the result to agents/TUI/API.

For read-only commands like `status`, `diff`, `log`, and `show`, the sidecar can run concurrently because it does not mutate the repo. For mutating commands like `commit`, `merge`, `rebase`, `reset`, `checkout`, `switch`, `push`, and `pull`, the sidecar should capture pre-state, let real Git mutate once, then capture post-state and act on the result. Running mutating Git commands twice is unsafe.

### What not to do

Do not implement the shell as:

```bash
git "$@" & jeryu git "$@"
```

That will recurse under the README shim and will double-run mutations. The shim should continue to route `git` to `jeryu git`, but `jeryu git` itself should internally run real Git exactly once and JeRyu sidecar work separately. The current shim makes this especially important because `git()` is defined as `command jeryu git "$@"`.

## High-priority issues to fix

### 1. `jeryu git` is currently only passthrough

Current implementation:

```rust
pub fn execute_git_passthrough(args: &[String]) -> Result<()> {
    let git_path = std::env::var("JERYU_SYSTEM_GIT").unwrap_or_else(|_| "/usr/bin/git".into());
    let status = std::process::Command::new(&git_path).args(args).status()?;
    std::process::exit(status.code().unwrap_or(1));
}
```

That preserves some compatibility, but it does not deliver the internal JeRyu value layer. It should become a Git command orchestrator that records command events, classifies command risk, captures before/after repo state, triggers validation, and updates the TUI/API. 

### 2. Hard-coded `/usr/bin/git` will break portability

`commands/git.rs` defaults to `/usr/bin/git`, while `shadow.rs` frequently uses `Command::new("git")`. This is inconsistent and brittle, especially for macOS, Homebrew, Nix, Windows/WSL, and custom Git installs.  

Fix: centralize Git resolution in one module:

```rust
src/git/runner.rs
```

Resolution order should be:

1. `JERYU_SYSTEM_GIT`, if set and not pointing to `jeryu`.
2. Installer-recorded path in `~/.jeryu/settings.json`.
3. `command -v git` captured before the shell shim is installed.
4. Known fallbacks only as last resort.

The installer should write the resolved Git path into settings or export `JERYU_SYSTEM_GIT` before defining the shell function.

### 3. Native Git wrappers swallow failures

`execute_save`, `execute_sync`, and `execute_ship` do not consistently propagate failure. For example, `execute_save` does not check `git add .` before committing; `execute_sync` returns `Ok(())` even if pull or push fails; and `execute_ship` prints success for remote push even though the origin push status is not checked before continuing. 

Fix: all wrappers must return the correct exit code. These commands are developer trust boundaries.

Suggested behavior:

```text
jeryu save
  git add .
  if add fails: exit add code
  git commit -m <message>
  exit commit code

jeryu sync
  git pull --rebase
  if pull fails: exit pull code
  git push
  exit push code

jeryu ship
  deprecated alias for: jeryu git push + wait/observe proof
```

### 4. The “shadow” story conflicts with the new product direction

The README, CLI, dispatch, serve lifecycle, settings, docs, and TUI still speak in terms of `shadow` remotes or shadow sync. That made sense for an earlier internal mirroring model. It now creates confusion because the better model is “every Git command becomes observable and agent-operable.”     

Fix: keep shadow compatibility for one release, but hide/deprecate it publicly.

Rename conceptually:

```text
shadow remote        -> mirror target
shadow sync          -> Git sidecar mirror
jeryu ship           -> jeryu git push + proof watch
ShadowSettings       -> GitSettings / MirrorSettings
shadow_sync_configs  -> git_mirror_targets or legacy_shadow_sync_configs
```

### 5. `dispatch.rs` has too much business logic

The file says it wires commands to modules and has “no business logic,” but it contains substantial command behavior for serve, shadow commands, action listing, `next`, and `explain-blocker`. The package metadata also says `main.rs` should be dispatcher only and business logic should live in modules.  

Fix: make `dispatch.rs` boring. It should call:

```rust
commands::git::execute(...)
commands::shadow::execute(...)
commands::next::execute(...)
commands::explain_blocker::execute(...)
commands::serve::execute(...)
commands::action::execute(...)
```

This makes the codebase easier for agents to patch safely.

### 6. `main.rs` has registry setup that should move out

`main.rs` contains a large `witness_rt::register_cells(...)` block. It is initialization, but it makes main noisy and undermines the “dispatcher only” invariant. Move it to:

```rust
src/witness_registry.rs
// or
src/witness/mod.rs::register_all_cells()
```

Then `main.rs` becomes:

```rust
jeryu::witness::register_all_cells();
jeryu::settings::init()?;
dispatch::run(cli).await
```

The current main path initializes tracing, registers witness cells, initializes settings, and delegates to dispatch. 

### 7. Supersedence is typed but underused

`decision.rs` defines `SupersedenceAction::{Cancel, Preserve, Degrade, Ignore}`, but `engine.rs` always creates `SupersedenceAction::Cancel` for older pending/running/created pipelines on the same ref. That misses your own roadmap’s idea of partial reuse, degradation, and preservation.  

Fix: add a real policy function:

```rust
pub fn decide_supersedence(input: SupersedenceInput) -> SupersedenceDecision
```

Rules:

```text
Cancel    old full/integration jobs made obsolete by new head
Preserve  completed evidence still valid for unchanged impact graph
Degrade   expensive non-gating jobs after newer commit arrives
Ignore    release/prod/protected/manual jobs that must not auto-cancel
```

### 8. Push hook branch-creation handling is incomplete

In `engine.rs`, branch creation with all-zero `before` is logged as “semantic evaluation bypassed,” but the function still continues into supersedence and impact planning using the all-zero before SHA. That can produce noisy or incorrect impact behavior. 

Fix:

```rust
if payload.before == ZERO_SHA {
    record_branch_created_event(...);
    maybe_plan_from_merge_base_or_default(...);
    return;
}
```

Or explicitly route to a branch-create policy.

### 9. Capability API needs Git-native intents

The current capability API supports patch proposal, race patches, tests, capsules, merge requests, blockers, snapshots, action listing, and validation planning. It does not yet expose Git command events or Git-sidecar control, which will be essential if agents are meant to operate JeRyu as a Git replacement. 

Add intents:

```rust
AgentIntent::GetGitSnapshot { repo_root: Option<String> }
AgentIntent::GetGitCommandEvent { event_id: i64 }
AgentIntent::ExplainGitCommand { event_id: i64 }
AgentIntent::ApproveGitRisk { request_id: String, grant: CapabilityGrantProof }
AgentIntent::TriggerValidationForGitEvent { event_id: i64, scope: String }
```

## API improvements

### Add a Git observability API

Create one shared Rust API consumed by CLI, TUI, MCP, and capability server:

```rust
pub struct GitObservabilityApi {
    db: Db,
}

impl GitObservabilityApi {
    pub async fn recent_events(&self, repo: Option<&Path>, limit: usize) -> Result<Vec<GitCommandEventView>>;
    pub async fn snapshot(&self, repo: Option<&Path>) -> Result<GitSnapshot>;
    pub async fn explain(&self, event_id: i64) -> Result<GitCommandExplanation>;
    pub async fn approve_risk(&self, request: GitRiskApproval) -> Result<GitRiskApprovalReceipt>;
}
```

### Add structured explanations

For every Git command, JeRyu should be able to answer:

```text
What did the user/agent ask Git to do?
Did real Git succeed?
What changed in HEAD, branch, index, worktree, remotes?
What proof did JeRyu trigger?
What is blocked?
What should happen next?
```

This directly supports the existing roadmap primitives: evidence capsules, risk gates, impact graph, confidence gate, and structured causal explanations. 

### Add durable nonce/replay protection

`capability.rs` tracks nonces in process memory with a `OnceLock<Mutex<HashSet<String>>>`. That prevents replay only until restart and is not enough for serious agent governance. Store nonces/idempotency keys in the DB with expiry.

## The product principle I would enforce everywhere

**JeRyu must never surprise a developer about what Git did. It should only add intelligence around Git.**

That means:

```text
Git exit status remains truth for the foreground command.
JeRyu sidecar status is additional evidence.
Humans can keep working even if sidecar is degraded.
Agents must obey sidecar policy before risky writes.
Every command is explainable after the fact.
```

That is the path to making this feel like the #1 AI-ready Git replacement: not “Git but with magic,” but **Git with a durable brain, proof system, agent API, and live mission control.**
