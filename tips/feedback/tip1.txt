I studied the repo through GitHub/API inspection. The biggest strategic finding is exciting but very clear: **JeRyu’s docs and product story already point toward “Git-compatible AI-era version control,” but the current implementation of `jeryu git …` is still only a thin system-Git passthrough.** The next level is not deleting everything called “shadow” immediately; it is replacing “shadow Git” with a **Git Command Mirror**: every Git command becomes a structured, observable, policy-aware JeRyu event while system Git remains compatibility-perfect.

## Executive diagnosis

JeRyu already has a strong foundation: a Rust single-binary control plane, GitLab runner orchestration, state storage, a Ratatui TUI, action registry, capability/MCP surfaces, VTI, failure evidence, and agent workflows. The README’s public promise says JeRyu wraps Git first, then replaces it, preserving `git` muscle memory through `jeryu git`, native wrappers, and `jeryu ship` pushing to both origin and a shadow CI remote. 

But the implementation has not yet caught up to that promise. `cli.rs` defines `Git { args }` as a passthrough command, and `dispatch.rs` routes it directly to `commands::git::execute_git_passthrough(&args)`.   That function currently just resolves `JERYU_SYSTEM_GIT` or defaults to `/usr/bin/git`, invokes Git with the provided args, and exits with Git’s status. It does **not** run JeRyu-side orchestration, does **not** mirror commands, does **not** record a structured event, and does **not** trigger runners or agents. 

So my strongest recommendation is:

> **Make `jeryu git …` the root product primitive. Everything else should hang off a typed Git command event ledger.**

That lets you keep the magic shell shim, preserve exact Git behavior, and make every normal Git command a launch point for JeRyu’s AI/CI/evidence machinery.

# Core architecture change: Command Mirror, not “shadow git”

Your idea is directionally right: agents and users should not need to remember separate “shadow” commands. But the safe implementation should not literally and blindly run every command as:

```bash
git <command> &
jeryu git <command> &
```

That can recurse, duplicate mutations, and create divergence. The right primitive is:

```text
user types: git <args>
shell shim calls: jeryu git <args>

JeRyu does:
1. run real system git with exact compatibility
2. in parallel or causally-after-success, run JeRyu sidecar behavior
3. record both outcomes in a durable event ledger
```

The distinction matters. For read-only commands, true parallel sidecar work is safe. For destructive or ref-mutating commands, JeRyu must preserve Git semantics and avoid creating a state where origin failed but JeRyu’s internal mirror succeeded.

I would introduce three mirror modes:

| Mode            | Behavior                                                               | Best for                                                  |
| --------------- | ---------------------------------------------------------------------- | --------------------------------------------------------- |
| `observe`       | Run system Git only; record JeRyu event and update TUI/state.          | Default for unknown/destructive commands.                 |
| `after-success` | Run Git first; if Git succeeds, trigger JeRyu mirror/CI/agent sidecar. | Safe default for `push`, `commit`, `merge`, `rebase`.     |
| `parallel`      | Run Git and JeRyu sidecar concurrently; reconcile outcomes.            | Read-only commands and explicit fast-path push workflows. |

This gives you the “all Git commands automatically activate JeRyu” behavior without corrupting repositories or surprising users.

# Critical issues to fix immediately

## 1. `jeryu git …` is not currently dual-run

Current behavior is just:

```rust
let git_path = std::env::var("JERYU_SYSTEM_GIT").unwrap_or_else(|_| "/usr/bin/git".into());
let status = std::process::Command::new(&git_path).args(args).status()?;
std::process::exit(status.code().unwrap_or(1));
```

That is compatibility-only, not JeRyu-aware. 

**Fix:** create a `DualGitExecutor` that runs system Git and a JeRyu sidecar event path. Preserve system Git’s stdout, stderr, exit status, and behavior by default. JeRyu metadata should not pollute stdout unless explicitly requested, because scripts depend on Git output.

## 2. `execute_ship` can claim success even when origin push fails

`execute_ship` runs `git push origin HEAD` but does not check whether that push succeeded before printing success-ish output after the shadow push attempt. 

That is a P0 trust bug. Shipping must fail loudly if origin fails.

**Fix:** make origin push canonical. In default mode:

```text
origin push fails  -> command fails, no JeRyu mirror push unless explicit --parallel/--force-mirror
origin succeeds    -> JeRyu mirror/runner trigger may run
JeRyu sidecar fails -> Git exit remains successful, but JeRyu records sidecar failure and TUI shows divergence
```

## 3. `save`, `sync`, and `undo` swallow failures

`execute_save`, `execute_sync`, and `execute_undo` return `Ok(())` even when underlying Git operations fail, except for process-spawn errors. 

**Fix:** all Git wrapper commands should return a typed outcome:

```rust
pub struct GitCommandOutcome {
    pub git_exit_code: i32,
    pub jeryu_sidecar_status: SidecarStatus,
    pub event_id: Option<i64>,
}
```

Then `main` exits with the right code.

## 4. Hard-coded `/usr/bin/git` is fragile

The installer asks users to add a shell function that routes `git` to `jeryu git`, but the runtime fallback is `/usr/bin/git`.  That is brittle on macOS, Homebrew installs, Nix, Windows/MSYS, custom Git locations, and any environment where `/usr/bin/git` is a shim.

**Fix:** during install, resolve the real Git path **before** installing the shell shim and write it to `~/.jeryu/settings.json` or export `JERYU_SYSTEM_GIT`. Also add runtime recursion protection so JeRyu never resolves itself as Git.

## 5. `std::process::exit` inside `commands::git` blocks cleanup and testing

`execute_git_passthrough` exits from inside a command module.  This prevents dispatch/main from doing cleanup, telemetry flushing, and structured error handling.

**Fix:** command modules should return `Result<ExitCode>` or `Result<i32>`. Let `main` be the only place that exits.

## 6. Existing shadow sync should be deprecated, not deleted immediately

`shadow.rs` is still wired into `serve`; `dispatch.rs` spawns `shadow::run_shadow_loop`, and `shadow.rs` owns remote mirroring/status/push logic.   Removing it immediately would break documented behavior and TUI summary state.

**Fix:** freeze it as a compatibility layer, then migrate it into the new Git Command Mirror. Rename the product concept from “shadow” to “JeRyu mirror” or “AI validation remote.”

## 7. `shadow.rs` has security and correctness problems

`shadow.rs` shells out to plain `git`, not the configured system Git. It force-pushes with `+...:refs/heads/...`, optionally pushes to an upstream URL, and embeds a PAT into an HTTP push URL. 

Problems:

* It may use the wrong Git binary.
* Force-push semantics need explicit policy.
* PAT-in-argv can leak via process listings.
* Upstream and shadow outcomes can diverge.
* There is no first-class event ledger tying mirror actions back to the user’s original Git command.

**Fix:** move all Git process execution through one `git::system` layer, avoid secrets in process args where possible, and write every mirror action to a `git_command_events` / `git_mirror_jobs` ledger.

# Proposed implementation: new Git Event Plane

Add a new module tree:

```text
src/git/
```

Keep `src/commands/git.rs`, but make it a thin CLI wrapper.

## `src/git/system.rs`

Responsibilities:

* Resolve the real Git binary.
* Prevent recursion through the shell shim.
* Support `JERYU_SYSTEM_GIT`.
* Support settings-based path.
* Validate with `git --version`.
* Provide one canonical function for all code that shells out to Git.

Suggested API:

```rust
pub struct SystemGit {
    pub path: PathBuf,
}

impl SystemGit {
    pub fn resolve(settings: &Settings) -> anyhow::Result<Self>;
    pub async fn run(&self, cwd: &Path, args: &[String]) -> anyhow::Result<GitProcessOutput>;
}
```

Files to update:

* `src/commands/git.rs`
* `src/shadow.rs`
* `install.sh`
* `src/settings.rs`

## `src/git/classify.rs`

Classify commands so JeRyu knows how to mirror them safely.

```rust
#[derive(Debug, Clone, Copy, Serialize, Deserialize)]
pub enum GitCommandClass {
    ReadOnly,
    LocalMutation,
    RefMutation,
    NetworkRead,
    NetworkWrite,
    Destructive,
    Unknown,
}
```

Examples:

* `status`, `log`, `show`, `diff`, `branch --list`: `ReadOnly`
* `add`, `commit`, `restore`, `checkout`, `switch`: `LocalMutation`
* `merge`, `rebase`, `reset`: `RefMutation` or `Destructive` depending args
* `fetch`, `ls-remote`: `NetworkRead`
* `push`: `NetworkWrite`
* `clean -fdx`, `reset --hard`, `rebase --abort`: `Destructive`

Unknown commands should still pass through to Git, but JeRyu should run sidecar in `observe` mode only.

## `src/git/event.rs`

Create the canonical event model:

```rust
#[derive(Debug, Serialize, Deserialize)]
pub struct GitCommandEvent {
    pub request_id: String,
    pub cwd: PathBuf,
    pub args: Vec<String>,
    pub class: GitCommandClass,
    pub mirror_mode: MirrorMode,
    pub actor: GitActor,
    pub started_at: DateTime<Utc>,
    pub before: GitSnapshot,
    pub after: Option<GitSnapshot>,
    pub git_exit_code: Option<i32>,
    pub sidecar_status: SidecarStatus,
}
```

This is the heart of the product. The event ledger turns normal Git usage into agent-visible intent.

## `src/git/executor.rs`

This should replace the direct passthrough.

Behavior:

1. Build event with command classification.
2. Capture pre-command HEAD/branch/remotes when safe.
3. Run system Git and sidecar according to mirror mode.
4. Preserve stdout/stderr/exit code.
5. Record post-command snapshot.
6. Trigger VTI/runners/agents where appropriate.
7. Return the system Git exit code.

Important: do not print JeRyu banners by default. The shell shim must be safe for scripts.

## `src/git/mirror.rs`

Responsibilities:

* Decide what JeRyu does for each command.
* On `push`, trigger internal validation.
* On `commit`, record new commit metadata and maybe precompute VTI plan.
* On `status`, optionally enrich TUI/state without stdout changes.
* On `merge/rebase`, record before/after topology.
* On `tag`, record release intent candidate.
* On `clone`, optionally register repo with JeRyu.

This can initially call existing `shadow` functionality behind the scenes. Later, it should fully replace the shadow loop.

## `src/git/policy.rs`

For agents, policy matters. A human typing `git reset --hard` can pass through. An agent doing the same should require explicit authority.

Policy inputs:

* actor type: human shell, MCP agent, capability agent, CI runner
* command class
* repo path
* branch/ref
* configured trust tier
* destructive args
* whether command writes to origin or JeRyu mirror

Outputs:

* allow
* allow but observe only
* require grant
* deny
* ask confirmation, for interactive human TTY only

# Simplifications without loss of extensibility

## 1. Collapse `ship`, `shadow push`, and Git push mirror into one engine

Today:

* `execute_ship` has separate origin/shadow push logic.
* `shadow_remote push` has separate push logic.
* `shadow_loop` has separate sync logic.

Future:

* all push-like behavior goes through `git::mirror`.

This reduces bugs and makes future policies easier.

## 2. Make action registry the one control surface

The action registry is already a strong design.  Extend it to Git actions instead of scattering permissions across CLI/TUI/MCP.

## 3. Move TUI screenshot/PNG code out of `mod.rs`

This does not remove functionality. It makes the TUI easier to maintain and test.

## 4. Keep GitLab-first, but stop making GitLab sound like the product

The product is JeRyu’s Git/agent command plane. GitLab is the first execution substrate. This aligns with the current mission’s “GitLab is the first infrastructure substrate, not the product itself.” 

## 5. Treat “shadow” as an implementation detail

The user should think:

```text
Git command -> JeRyu mirror -> proof/agents/runners
```

Not:

```text
I must separately manage shadow Git.
```

# The key product decision

The winning move is this:

> **Do not ask users or agents to learn a new workflow first. Make normal Git commands become JeRyu events automatically.**

That means `git push` should still feel like `git push`, but behind the scenes JeRyu should know:

* who pushed
* what changed
* which refs moved
* what validation is required
* what can be skipped safely
* which agent can help
* which runner should start
* what evidence proves the result
* whether origin and JeRyu diverged

Once that works, JeRyu stops being “a GitLab control plane with a Git wrapper” and becomes what the README is already reaching for: **the Git-compatible version control layer for the AI era.**
