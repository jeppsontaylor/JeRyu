## Executive take: JeRyu has the right bones, but the core product loop needs to move from “GitLab control plane with a Git shim” to “Git-compatible dual-run proof engine.”

The README already says the right thing: **“JeRyu wraps Git first, then replaces it,”** with a shell shim that routes `git ...` to `jeryu git ...`, plus `jeryu ship` for origin + shadow remote pushes. ([GitHub][1]) The current code, though, does **not** yet implement the big idea you described: automatically running every Git command through both the real Git path and the JeRyu/shadow/proof path. Today, `jeryu git ...` is essentially a passthrough to system Git, and `ship` is a special-case sequential push wrapper. 

That is the biggest upgrade: make **dual Git execution + proof receipts** the center of JeRyu.

I could not run a full local clone/build/test from this sandbox because direct `git clone` resolution was unavailable here, so I inspected the repository through GitHub pages and repository file APIs. The feedback below is based on the README, MISSION, Rust source, TUI, CLI dispatch, settings, shadow sync, and capability API surfaces I was able to review.

---

Drop this into `MISSION.md`, the README hero section, and `src/lib.rs` crate docs:

> **JeRyu’s mission is to make Git the ledger, and JeRyu the intelligent operating layer around it.**
>
> Every command a developer already knows should keep working. Behind that familiar interface, JeRyu mirrors critical Git actions, records proof receipts, routes work to isolated runners, and gives autonomous agents bounded, auditable power to test, repair, explain, and ship software.
>

>

Even shorter product tagline:

> **Git-compatible for humans. Agent-native by design. Proof-backed for every action.**

This better matches the README’s Git-compatible ambition while preserving the existing Mission document’s focus on replacing brittle human-babysat Git/GitLab processes with an agent-governed Rust control plane. ([GitHub][1]) ([GitHub][2])

# The most important architecture change

## Build a first-class `git` subsystem, not a passthrough command

Current state:

```rust
pub fn execute_git_passthrough(args: &[String]) -> Result<()> {
    let git_path = std::env::var("JERYU_SYSTEM_GIT").unwrap_or_else(|_| "/usr/bin/git".into());
    let status = std::process::Command::new(&git_path).args(args).status()?;
    std::process::exit(status.code().unwrap_or(1));
}
```

That means `jeryu git status` only delegates to system Git. It does not classify the command, mirror it, record proof, run a shadow path, or trigger JeRyu runners. 

Create a new subsystem:

```text
src/git/
  mod.rs
  command.rs       # GitCommand, cwd/env/stdin/tty metadata
  classify.rs      # read/write/network/destructive/interactive classification
  executor.rs      # exact system Git execution, output preservation
  fanout.rs        # primary + shadow/JeRyu execution policy
  policy.rs        # per-command fanout rules
  receipt.rs       # DualRunReceipt model
  store.rs         # receipt persistence
  redact.rs        # token/credential redaction
  shim.rs          # shell shim install/doctor helpers
  errors.rs
```

Then change `src/commands/git.rs` from a thin wrapper into:

```rust
pub async fn execute_git(args: Vec<OsString>, opts: GitRunOpts) -> Result<GitExit> {
    let command = GitCommand::capture(args, opts.cwd)?;
    let class = classify_git_command(&command)?;
    let policy = GitPolicy::load()?.plan(&command, &class)?;

    let receipt = fanout::run(command, class, policy).await?;
    receipt.print_user_facing_summary();

    std::process::exit(receipt.primary_exit_code());
}
```

The invariant should be:

> **Primary Git behavior is sacred. JeRyu may observe, mirror, prove, or accelerate, but default stdout, stderr, working tree mutation, and exit code must match system Git.**

That is how JeRyu becomes safe enough to replace `git` without scaring developers.

# What “run all Git commands in both places” should mean safely

Running every Git command twice in the same working tree would be dangerous. `git checkout`, `git reset`, `git merge`, `git rebase`, `git stash`, `git clean`, and `git commit` mutate local state. Running them twice can corrupt expectations, duplicate effects, or produce misleading output.

Instead, JeRyu should implement **safe dual execution by command class**.

## Command classes

### 1. Read-only local commands

Examples:

```text
status
log
show
diff
rev-parse
branch --show-current
remote -v
config --get
```

Default behavior:

```text
primary: run system git, stream output exactly
jeryu: record receipt + optionally enrich TUI/API state
shadow: no side effect
```

These are good candidates for parity checking later, but day one should preserve primary output exactly.

### 2. Local mutation commands

Examples:

```text
add
commit
checkout
switch
restore
reset
merge
rebase
cherry-pick
stash
tag
branch
clean
```

Default behavior:

```text
primary: run system git in the real worktree
jeryu: record before/after HEAD/index/worktree facts
shadow: mirror/replay after success into isolated shadow clone/worktree, not the same directory
```

Do **not** run these in parallel against the user’s working tree.

### 3. Network read commands

Examples:

```text
fetch
pull
ls-remote
submodule update
```

Default behavior:

```text
primary: run system git
jeryu: record refs before/after
shadow: optionally update shadow clone after primary success
```

`pull` should be treated as `fetch + merge/rebase`, so it belongs in a more cautious policy bucket.

### 4. Network write commands

Examples:

```text
push
push --tags
push --mirror
```

This is where parallel fanout is valuable:

```text
primary: push to origin/upstream remote
shadow: push same ref to JeRyu shadow remote/local GitLab
jeryu: record both outcomes, trigger runners, surface divergence
```

Default exit code should still be the **primary Git exit code** unless the repo explicitly opts into `block_on_shadow_failure = true`.

### 5. Interactive/destructive commands

Examples:

```text
commit       # without -m/-F in a TTY
rebase -i
add -p
checkout -p
clean -fdx
reset --hard
push --force
filter-branch
```

Default behavior:

```text
primary: run system git
jeryu: receipt only, with high-risk classification
shadow: disabled unless explicit policy enables it
```

JeRyu should not mirror these blindly.

## 2. `execute_ship` can report success when origin push failed

Current `execute_ship` runs:

```text
git push origin HEAD
git push shadow HEAD
```

But it does not verify the origin push status before saying the operation shipped. Shadow failure is explicitly tolerated, which may be fine, but primary failure should never print a success path. 

**Action**

Make `ship` a thin alias over the new fanout engine:

```text
jeryu ship == jeryu git push --jeryu-policy=parallel-network-write
```

or:

```text
git push
# through shell shim, automatically fans out to origin + shadow
```

**Required semantics**

```text
origin success + shadow success -> success
origin success + shadow failure -> primary success, visible warning, receipt records divergence
origin failure + shadow success -> failure, never say shipped
origin failure + shadow failure -> failure
```

**Tests**

```text
ship_origin_fail_returns_failure
ship_shadow_fail_warns_but_primary_exit_preserved
ship_origin_and_shadow_run_concurrently
ship_redacts_credentials_in_receipt
```

## 3. `save`, `sync`, and `undo` ignore important failures

`execute_save` does not check `git add .` status before committing. `execute_sync` does not return an error on failed push. `execute_undo` silently returns `Ok(())` even if reset fails. 

**Action**

Either delete these wrappers and re-express them through the Git engine, or make them strict:

```rust
run_git(["add", "."])?;
run_git(["commit", "-m", message])?;
```

`sync` should become:

```text
git pull --rebase
git push
```

through the same receipt/fanout path.

```text
save_fails_when_commit_fails

sync_fails_when_push_fails

```

## 4. Git execution is scattered across the codebase

`commands/git.rs` uses `JERYU_SYSTEM_GIT` with `/usr/bin/git` fallback. `shadow.rs` calls `"git"` directly. `commands/test.rs` calls `"git"` directly for `log`, `diff`, and `hash-object`. `system.rs` separately invokes Git for status.    

That fragmentation will make Git replacement impossible.

**Action**

All Git invocations should go through `src/git/executor.rs`.

```rust
let git = GitBinary::resolve(settings.git.system_git.as_deref())?;
git.run(GitInvocation::new(args).cwd(repo_root)).await?;
```

**Also fix**

Do not default to `/usr/bin/git`. That breaks macOS Homebrew paths, Nix, mise/asdf-managed Git, Windows compatibility, and container images. Resolve in this order:

```text
JERYU_SYSTEM_GIT
settings.git.system_git
PATH lookup for `git`
hard failure with `jeryu doctor git` suggestion
```

# P0/P1 security and agent-readiness fixes

## 6. Capability grants are modeled but not truly enforced

The capability API has a strong-looking envelope: protocol version, request ID, actor, nonce, expiry, budget, grant, and typed `AgentIntent`.  But the validation path currently checks mostly protocol/nonce/expiry shape. It does not appear to enforce that a mutating action has a valid grant matching the requested action, project, ref, path, head SHA, budget, and expiry. 

That is a blocker for “AI-ready Git replacement.”

**Action**

Add a proper policy gate:

```rust
pub fn validate_grant(
    request: &AgentActionRequest,
    action: &ActionEntry,
    resolved_scope: &ResolvedActionScope,
    db: &Db,
) -> Result<GrantDecision>
```

**Rules**

```text
Read-only actions: no grant required.
Local state / Git write / CI execution: require agent_task grant.
Merge: require merge_approval grant.
Production: require production_approval grant.
Grant action_id must match registry action id.
Grant project_id/ref/path scope must include requested mutation.
Grant head_sha must match current branch head when supplied.
Expired grant denied.
Replay nonce denied.
Budget exceeded denied.
Legacy requests denied by default.
```

**Tests**

```text
capability_denies_git_write_without_grant
capability_denies_wrong_action_grant
capability_denies_wrong_ref_scope
capability_denies_path_escape
capability_denies_expired_grant
capability_denies_replayed_nonce
capability_allows_readonly_without_grant
legacy_capability_denied_when_disabled
```

```rust
pub struct FileModification {
```

to:

```rust
pub enum FileModification {

    Update { path: String, content: String },
```

```text

deny symlink target escapes
deny oversized files unless grant allows
deny binary blobs unless explicit

```

# Shadow remote and dual-write fixes

## 9. Shadow sync should be folded into the Git fanout layer

`src/shadow.rs` already has useful pieces: remote status, ensure remote, push remote, background sync loop, summary computation.  Keep those, but make them subordinate to the Git dual-run engine.

Today there are two overlapping ideas:

```text
jeryu git ... passthrough
jeryu shadow ... background sync
jeryu shadow-remote ...
jeryu ship ...
```

That is too many ways to express the same product promise.

**Recommended product model**

```text
git push                 # through shim, fans out to origin + shadow
jeryu git push           # same behavior explicitly
jeryu shadow status      # inspect mirror/proof state
jeryu shadow repair      # repair divergence
jeryu shadow pause       # pause mirror fanout
jeryu shadow resume      # resume mirror fanout
```

Keep `shadow-remote ensure/status/push`, but make it a low-level admin command.

## 10. Do not put PATs in Git push URLs

`sync_once` injects the GitLab PAT into an HTTP URL:

```text
http://oauth2:{pat}@...
```

Then passes that URL to `git push`.  That risks exposing credentials through process listings, debug logs, receipts, or panic output.

**Action**

Use one of:

```text
GIT_ASKPASS helper script with env token
credential.helper scoped to temp config
extraHeader for HTTP auth
SSH deploy key
```

Also add redaction everywhere:

```rust
redact_git_url("https://oauth2:token@example/repo.git")
// https://oauth2:***@example/repo.git
```

**Tests**

```text
shadow_push_does_not_put_pat_in_argv
receipt_redacts_http_basic_token
logs_redact_oauth2_token
tui_redacts_token_in_remote_url
```

## 11. Avoid implicit force-push to shadow

`sync_once` builds a refspec with `+`:

```text
+HEAD:refs/heads/<target_branch>
```

That allows non-fast-forward updates.  For a shadow mirror this may be intentional sometimes, but it should be explicit policy, not default.

**Action**

Default:

```text
HEAD:refs/heads/<target_branch>
```

Only use `+` when:

```text
settings.git.shadow.allow_force = true
```

or when the original command itself was a force push and policy allows shadow force.

## Make the action registry executable

```rust

    pub label: &'static str,

}
```

to include:

```rust
pub struct ActionEntry {
    pub id: &'static str,
    pub label: &'static str,
    pub risk_tier: RiskTier,
```

```rust
impl Drop for TerminalGuard {

            crossterm::event::DisableMouseCapture

}
```

# The crisp product promise

The phrase I would build everything around:

> **Keep typing Git. JeRyu makes it self-driving.**

The implementation promise:

> **For every Git command, JeRyu preserves primary Git behavior, records proof, mirrors what is safe, blocks what is unsafe, and exposes structured state to humans, runners, and agents.**

That is the path from the current repo to a serious “#1 AI-ready Git replacement.”
